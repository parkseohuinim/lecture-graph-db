# LLM과 Graph DB로 구축하는 지능형 지식 그래프(온톨로지) 입문

## 강의 개요

> **"수사 보고서 몇 장을 넣었을 뿐인데, AI가 범인을 찾아냈다."**
>
> 이 강의에서는 여러분이 직접 그런 시스템을 처음부터 만들어 봅니다. 프로그래밍 경험이 조금이라도 있다면 충분합니다. 나머지는 이 강의에서 하나하나 함께 해봅니다.

| **목표**   | 비정형 텍스트에서 LLM으로 지식 그래프를 자동 구축하고, 자연어 질의로 범인을 추적하는 수사 프로파일링 시스템 완성 |
| **도메인** | 팔란티어 Gotham 스타일 — 수사 프로파일링 & 범인 추적         |

### 이런 분들에게 추천합니다

- "그래프 데이터베이스"라는 말을 들어봤지만, 정확히 뭔지 모르겠는 분
- ChatGPT는 써봤지만, API로 내 프로그램에 연결해 본 적은 없는 분
- 데이터베이스라고 하면 MySQL/PostgreSQL만 써본 분
- "지식 그래프", "온톨로지"가 뭔지 궁금했던 분
- 뭔가 **눈에 보이는 결과물**을 만들면서 배우고 싶은 분

### 이 강의를 마치면 할 수 있는 것

1. **Neo4j**(그래프 데이터베이스)를 설치하고, 데이터를 넣고 꺼낼 수 있다
2. **Cypher**(그래프 전용 쿼리 언어)로 복잡한 관계를 한 줄로 탐색할 수 있다
3. **LLM(ChatGPT) API**를 호출해서 비정형 텍스트에서 구조화된 정보를 자동 추출할 수 있다
4. **FastAPI**로 백엔드 서버를 만들어 위 기능들을 REST API로 제공할 수 있다
5. "누가 범인인가?"라는 질문에 **AI + 그래프 분석**으로 근거 기반 답변을 생성할 수 있다

## 세션 1 — 왜 그래프인가: 팔란티어가 보여준 것

> **유형**: 이론
>
> 이 세션에서는 코딩 없이 "왜 그래프 데이터베이스가 필요한가"를 이해합니다.
> 끝나면 "아, 그래서 그래프를 쓰는 거구나!"라는 감이 잡힐 거예요.

### 1.1 데이터베이스란 무엇인가 — 그래프 DB를 이해하기 위한 출발점

#### 일상에서 만나는 데이터베이스

여러분이 네이버에서 검색하거나, 쿠팡에서 주문하거나, 카카오톡으로 메시지를 보낼 때 — 이 모든 데이터는 **데이터베이스(DB)** 에 저장됩니다. 데이터베이스는 쉽게 말해 **"데이터를 체계적으로 저장하고, 원하는 데이터를 빠르게 찾아주는 소프트웨어"** 입니다.

엑셀 파일을 생각해 보세요. 엑셀도 데이터를 저장하지만, 사용자가 10만 명이 동시에 접속하면 망가집니다. 데이터베이스는 이런 상황에서도 안정적으로 돌아가도록 설계된 전문 소프트웨어입니다.

데이터베이스는 크게 두 가지 철학으로 나뉩니다:

**관계형 데이터베이스 (RDBMS) — "엑셀 표처럼 저장한다"**
- MySQL, PostgreSQL 같은 전통적 DB
- 데이터를 **테이블(표)** 형태로 저장합니다
  - **행(row)** = 데이터 한 건 (예: "김철수"라는 사람 한 명의 정보)
  - **열(column)** = 속성 (예: 이름, 나이, 주소)
- 테이블 간 관계를 표현하려면 **JOIN** 이라는 연산을 사용해야 합니다
- 쇼핑몰의 주문 내역, 은행의 계좌 정보처럼 **표 형태로 딱 떨어지는 데이터**에 강합니다

```
[사람 테이블]                    [장소 테이블]
| ID | 이름   | 나이 |          | ID | 이름   | 주소     |
|----|--------|------|          |----|--------|----------|
| 1  | 김철수 | 35   |          | 1  | 서울역 | 용산구   |
| 2  | 박영수 | 40   |          | 2  | 강남역 | 강남구   |
```

**그래프 데이터베이스 (Graph DB) — "점과 선으로 저장한다"**
- Neo4j, Amazon Neptune 같은 DB
- 데이터를 **노드(점)** 와 **관계(선)** 으로 저장합니다
- "누가 누구를 아는가", "어디서 누구를 만났는가" 같은 **관계 중심 질문**에 최적화
- 관계를 따라가는 탐색이 극도로 빠릅니다

```
(김철수)---[아는 사이]--->(박영수)
   |                         |
   [있었음]                 [있었음]
   |                         |
   v                         v
(서울역)                  (서울역)
```

> **비유: 전화번호부 vs 거미줄**
>
> 관계형 DB는 **전화번호부**와 비슷합니다. 이름 순으로 잘 정리되어 있어서 특정 사람을 찾기 쉽지만, "이 사람의 친구의 친구는 누구?"를 알려면 여러 페이지를 왔다갔다해야 합니다.
>
> 그래프 DB는 **거미줄**과 비슷합니다. 모든 것이 실(관계)로 연결되어 있어서, 실을 따라가기만 하면 연결된 모든 것을 쉽게 찾을 수 있습니다.

**핵심 차이: "관계"를 얼마나 중요하게 다루는가?**

관계형 DB에서 "관계"는 테이블 사이의 외래 키(FK, 다른 테이블의 ID를 참조하는 번호)로 간접 표현됩니다. 반면 그래프 DB에서 관계는 노드와 동등한 지위를 가진 독립적 객체입니다. 관계 자체에 속성(언제, 어디서, 어떻게)을 붙일 수 있고, 관계를 따라 탐색하는 것이 DB의 핵심 연산입니다.

### 1.2 관계형 DB의 한계와 그래프의 등장

자, 구체적인 상황을 하나 생각해 봅시다.

**상황**: 수사관이 용의자 A의 "3단계 인맥"을 조회하려 합니다.
- 1단계: 용의자 A가 직접 아는 사람
- 2단계: 그 사람들이 아는 사람
- 3단계: 그 사람들이 또 아는 사람

인스타그램의 "팔로워의 팔로워의 팔로워"를 찾는 것과 비슷합니다.

**관계형 DB 접근 — JOIN 지옥:**

```sql
-- 용의자 A의 3단계 인맥을 찾으려면...
-- 테이블을 3번이나 연결(JOIN)해야 합니다!
SELECT DISTINCT p3.name
FROM person p1
JOIN relationship r1 ON p1.id = r1.from_id      -- 1단계 연결
JOIN person p2 ON r1.to_id = p2.id
JOIN relationship r2 ON p2.id = r2.from_id      -- 2단계 연결
JOIN person p3 ON r2.to_id = p3.id
JOIN relationship r3 ON p3.id = r3.from_id      -- 3단계 연결
WHERE p1.name = '용의자 A';
-- 5단계, 10단계로 늘어나면? → 코드도 폭발하고, 성능도 폭발(느려짐)
```

**그래프 DB 접근 — 한 줄이면 끝:**

```cypher
MATCH (a:Person {name: '용의자 A'})-[*1..3]-(connected)
RETURN DISTINCT connected.name
// 끝! 5단계, 10단계로 바꾸고 싶으면 숫자만 바꾸면 됩니다.
// [*1..3] → [*1..10]
```

> **느끼셨나요?** 관계형 DB에서는 "단계"가 늘어날 때마다 코드가 점점 복잡해지고 느려지는 반면, 그래프 DB에서는 숫자 하나만 바꾸면 됩니다. 이것이 그래프 DB의 핵심 장점입니다.

**왜 이런 차이가 나는가? — 저장 방식의 근본적 차이**

쉽게 비유하면:

- **관계형 DB**: 전화번호부에서 사람을 찾고 → 그 사람의 친구 이름을 보고 → 다시 전화번호부에서 그 친구를 찾고 → 반복... (매번 처음부터 찾아야 함)
- **그래프 DB**: 각 사람이 친구 집 주소를 직접 알고 있음 → 바로 찾아감 → 그 친구도 자기 친구 집 주소를 알고 있음 → 바로 찾아감 (목록을 뒤질 필요 없음)

기술적으로는 이것을 "**Index-free Adjacency**(인덱스 없는 인접성)"라고 합니다. 관계형 DB는 매번 인덱스(색인)를 검색해야 하지만, 그래프 DB는 각 노드가 연결된 노드의 실제 주소를 직접 가지고 있어서 바로 찾아갈 수 있습니다.

| 관계 깊이 | 관계형 DB (JOIN)  | 그래프 DB (Traversal) |
| --------- | ----------------- | --------------------- |
| 1단계     | 빠름              | 빠름                  |
| 2단계     | 약간 느림         | 빠름                  |
| 3단계     | 눈에 띄게 느림    | 빠름                  |
| 5단계+    | 사실상 불가       | 여전히 빠름           |

**핵심 메시지**: 관계가 복잡하게 얽힌 데이터일수록 그래프 DB가 압도적으로 유리합니다.

### 1.3 팔란티어 Gotham 사례 분석

> **팔란티어(Palantir)** 는 미국의 데이터 분석 회사로, CIA와 FBI 같은 기관에 수사 분석 소프트웨어를 제공합니다. 그들의 핵심 기술이 바로 **그래프 기반 데이터 분석**입니다. 우리가 이 강의에서 만드는 시스템은, 팔란티어의 Gotham이라는 제품의 미니 버전이라고 생각하면 됩니다.

**팔란티어가 수사/정보 분석에서 하는 일:**

1. **이종 데이터 통합**: 통화 기록, 금융 거래, CCTV, 진술서, SNS → 하나의 그래프로 통합
2. **엔티티 해소 (Entity Resolution)**: "김철수" = "철수" = "용의자 A"를 동일 인물로 매핑
3. **관계 네트워크 분석**: 누가 누구와 연결되었나, 숨겨진 중간 매개자는 누구인가
4. **패턴 탐지**: 비정상적 금융 흐름, 반복적 접선 패턴, 시간대별 행동 궤적
5. **다단계 추적**: A → B → C → D로 이어지는 연결 고리에서 핵심 인물 식별

**그래프 분석이 실제로 쓰이는 분야:**

> **한 가지 먼저 알아두세요.** 아래 사례들은 "팔란티어가 이 사건을 해결했다"라기보다, **"그래프 방식의 사고가 왜 강력한가"**를 보여주는 맥락으로 이해해 주세요. 공개 보도로 사건 전모와 팔란티어의 직접 기여가 1:1로 확인되는 경우는 제한적입니다.

**1) 빈 라덴 추적 — 네트워크 추적의 교과서적 사례**

공개된 핵심 줄거리는, 미국 정보당국이 여러 해에 걸쳐 빈 라덴의 연락책(Abu Ahmed al-Kuwaiti)을 식별하고, 이 인물을 추적해 파키스탄 아보타바드 은신처로 이어졌다는 것입니다. 본질은 **"한 사람을 중심으로 연결망을 좁혀 가는 네트워크 추적"**이었습니다.

다만 이 사건을 "팔란티어가 은신처를 특정했다"라고 단정하기는 어렵습니다. 이 사례는 **그래프 방식으로 사람·전화·장소의 연결을 좁혀 가는 수사/정보분석 사고방식**을 보여주는 대표적 예시로 이해하는 것이 정확합니다.

그래프 관점에서 보면 구조는 이렇습니다: 사람, 전화번호, SIM, 기지국 위치, 차량, 주소를 각각 노드로 만들고, "통화했다 / 함께 이동했다 / 같은 장소에 반복 출현했다 / 동일 자산을 공유했다"를 엣지로 연결합니다. 처음에는 무수한 점과 선이지만, 시간이 지나면 **항상 특정 소수 인물에게 수렴하는 허브**가 드러납니다.

> 참고: 팔란티어가 대테러 분석 분야에 강점을 가진 회사인 것은 맞습니다. 실제로 프랑스 정보당국은 2015년 파리 테러 이후 팔란티어를 대테러 분석에 활용했다고 Reuters가 보도했습니다.

**2) 금융 사기 네트워크 — 팔란티어와 가장 분명하게 연결되는 분야**

이 분야는 팔란티어의 역할이 공개 자료로 가장 잘 확인됩니다. Reuters는 2014년 미국 SEC(증권거래위원회)가 팔란티어 플랫폼을 사용해 거래기록·개인 연락처·제보·민원·참고자료를 연결하여 내부자거래, 회계부정 등 불법행위를 탐지했다고 보도했습니다.

자금세탁은 한 건의 거래만 보면 평범해 보입니다. A사→B사→C사→해외계좌→A의 관계인... 개별 시스템에서는 각각 정상 송금이지만, 그래프로 올리면 **같은 주소·같은 대표자·같은 IP·짧은 시간 안의 반복 회전 송금**이 한꺼번에 엮이면서 여러 법인을 돌려쓰는 구조가 드러납니다.

팔란티어도 자사 AML(자금세탁방지) 문서에서 KYC, 거래정보, 제재 리스트, 법인등기, IP, 디바이스 ID 등을 통합해 **엔터티 해석(entity resolution) + 링크 분석(link analysis)**으로 순환 자금 흐름, 의심스러운 페이퍼컴퍼니, 실소유주 네트워크를 식별한다고 설명합니다.

핵심은, 팔란티어가 "한 번에 범인을 알아내는 마법"이 아니라, **서로 다른 시스템에 갇혀 있던 조각 정보들을 하나의 그래프로 끌어올려, 인간 조사관이 "이 회사들이 사실 같은 네트워크구나"라고 판단할 수 있게 만드는 조사 증폭기**라는 점입니다.

**3) 마약 조직 와해 — 장기적 수사 인프라로서의 그래프 분석**

이 분야는 "유명 단일 사건" 보다 **지속적 수사 인프라**로 이해하는 것이 정확합니다. 팔란티어는 2022년 보도자료에서 미국 국토안보수사국(HSI)을 지원하는 사건관리 소프트웨어가 인신매매, 마약 밀수, 초국가 범죄조직 수사를 지원한다고 밝혔습니다.

마약 조직은 보스–중간 브로커–지역 배포책–운반책–자금세탁 담당–합법 사업체의 다층 구조를 가지지만, 선불폰·차명 차량·현금만 사용하기 때문에 일반 표 형식 데이터로는 구조가 잘 보이지 않습니다. 통화기록, 기지국 위치, 차량 번호판, 국경 통과 기록, 계좌 입출금 등을 하나의 네트워크로 통합하면, **누가 허브인지, 어느 노드가 제거되면 조직 전체가 흔들리는지**가 드러납니다.

실무적으로 중요한 점은, 조직의 계층 구조가 "직급표"처럼 자동으로 그려지는 것이 아니라, **반복 접촉·공동 이동·공동 자산 사용** 같은 패턴에서 중심성과 역할이 드러난다는 것입니다. 이를 통해 수사기관은 말단만 잡는 것이 아니라 조직의 중간 허브와 자금 흐름 축을 동시에 끊을 수 있습니다.

---

**한 줄 요약:**

| 사례 | 그래프 분석의 역할 | 팔란티어와의 관계 |
| --- | --- | --- |
| 빈 라덴 추적 | 연락책 중심의 네트워크 수렴 | 그래프적 사고방식의 대표 사례 (직접 기여 단정은 어려움) |
| 금융 사기/자금세탁 | 페이퍼컴퍼니·순환 송금 구조를 네트워크로 식별 | 가장 분명 (SEC 활용 보도, 팔란티어 AML 공식 자료) |
| 마약 조직 와해 | 다층 조직의 허브·자금 흐름 축 식별 | HSI 등 장기적 수사 플랫폼으로 지원 |

> **"왜 엑셀이나 기존 DB로는 안 되나요?"**
>
> 위 사례에서 공통적인 것은 **"관계의 관계의 관계"** 를 추적해야 한다는 점입니다. 단순히 "누가 누구를 알아"가 아니라, "이 사람이 아는 사람이 아는 사람 중에 범인과 통화한 적이 있는 사람은?"처럼 여러 단계를 넘나드는 질문에는 그래프 DB가 필수적입니다. 그리고 팔란티어의 핵심 가치도 결국 이 지점 — **서로 떨어져 있던 데이터를 하나의 그래프 위에서 이어 붙여, 사람이 볼 수 없던 연결을 보게 만드는 것** — 에 있습니다.

### 1.4 온톨로지와 지식 그래프 핵심 개념

> 이 부분은 용어가 조금 어렵게 느껴질 수 있습니다. 하지만 핵심은 간단합니다. 차근차근 따라와 보세요.

**온톨로지(Ontology)란? — "세상을 어떻게 분류할 것인가"의 설계도**

온톨로지는 "이 세계에는 어떤 종류의 것들이 존재하고, 그것들 사이에는 어떤 관계가 가능한가"를 정의한 **설계도**입니다.

일상적인 비유로 설명하면:

- **동물원의 분류표**: "포유류에는 사자, 호랑이가 있고, 조류에는 독수리, 펭귄이 있다" — 이것이 동물원의 온톨로지
- **도서관의 분류 체계**: "소설, 과학, 역사, 철학..." — 이것이 도서관의 온톨로지
- **수사 도메인의 온톨로지**: "사람, 장소, 사건, 증거, 조직, 차량, 전화, 계좌가 존재하고, 이들 사이에 '아는 사이', '통화함', '있었음' 같은 관계가 가능하다"

프로그래밍을 해보신 분이라면, 온톨로지는 **클래스(class) 정의**에 해당한다고 생각하면 됩니다.

**지식 그래프(Knowledge Graph)란? — "설계도에 실제 데이터를 채운 것"**

지식 그래프는 온톨로지 설계도에 따라 **실제 데이터를 채워 넣은 결과물**입니다.

- 온톨로지: "사람(Person)이라는 타입이 있고, 장소(Location)에 '있었음(WAS_AT)' 관계로 연결될 수 있다" ← 설계도
- 지식 그래프: "**김철수**라는 Person이 **서울역**이라는 Location에 **3월 5일 22시**에 WAS_AT 관계로 연결" ← 실제 데이터

프로그래밍으로 비유하면:
- 온톨로지 = **클래스(class)** 정의 (`class Person`, `class Location`)
- 지식 그래프 = 클래스로부터 만들어진 **객체(object) 인스턴스**들과 그 관계 (`김철수 = Person(name="김철수")`)

**Property Graph 모델 — Neo4j가 데이터를 저장하는 방식**

이 강의에서 사용하는 Neo4j는 "Property Graph 모델"이라는 방식으로 데이터를 저장합니다. 세 가지 핵심 요소만 기억하면 됩니다:

- **노드(Node)**: "것" 하나를 나타냄 (예: 김철수라는 사람, 서울역이라는 장소)
  - **레이블(Label)**: 이 노드가 어떤 종류인지 분류하는 태그 (예: `Person`, `Location`)
  - **속성(Property)**: 이 노드의 구체적 정보 (예: `name: '김철수'`, `age: 35`)
- **관계(Relationship)**: 노드와 노드 사이의 연결선 (예: `KNOWS`, `WAS_AT`)
  - **방향**이 있습니다: (김철수)-[KNOWS]->(박영수)
  - 관계에도 **속성**을 붙일 수 있습니다: `since: 2020`

```
예시:
(김철수:Person {age:35}) ---[KNOWS {since:2020}]---> (박영수:Person {age:40})
 ^^^^^ ^^^^^^  ^^^^^^^^      ^^^^^ ^^^^^^^^^^^       ^^^^^ ^^^^^^  ^^^^^^^^
 이름  레이블   속성          관계   관계의 속성        이름  레이블   속성
```

**오늘 사용할 수사 도메인 온톨로지:**

| 노드 타입      | 속성 예시                     | 설명                      |
| -------------- | ----------------------------- | ------------------------- |
| `Person`       | name, alias, age, description | 용의자, 목격자, 피해자 등 |
| `Location`     | name, address, lat, lng       | 사건 현장, 목격 장소 등   |
| `Event`        | type, date, time, description | 사건, 목격, 통화, 거래 등 |
| `Evidence`     | type, description, source     | 물증, 디지털 증거 등      |
| `Organization` | name, type                    | 회사, 단체 등             |
| `Vehicle`      | plate, model, color           | 차량 정보                 |
| `Phone`        | number, carrier               | 전화번호                  |
| `Account`      | number, bank, owner           | 금융 계좌                 |

| 관계 타입         | 설명                         |
| ----------------- | ---------------------------- |
| `KNOWS`           | 인물 간 아는 사이            |
| `CALLED`          | 전화 통화                    |
| `WAS_AT`          | 특정 시간에 장소에 있었음    |
| `OWNS`            | 소유 관계 (차량, 계좌, 전화) |
| `TRANSFERRED_TO`  | 금전 이체                    |
| `WITNESSED`       | 목격                         |
| `PARTICIPATED_IN` | 사건 참여                    |
| `FOUND_AT`        | 증거 발견 장소               |
| `WORKS_FOR`       | 소속 조직                    |
| `RELATED_TO`      | 기타 관련성                  |

### 1.5 오늘의 빅픽처 — 우리가 만들 것의 전체 그림

지금까지 배운 개념을 조합하면, 우리가 만들 시스템의 전체 흐름이 보입니다.

**한 줄 요약**: 수사 보고서를 넣으면 → AI가 자동으로 관계도를 만들고 → "범인이 누구야?"라고 물으면 답해주는 시스템

```
[1단계] 수사 보고서/진술서를 입력합니다 (그냥 텍스트)
         ↓
[2단계] FastAPI(우리가 만드는 서버) + GPT(AI)가
         텍스트에서 인물, 장소, 관계 등을 자동으로 뽑아냅니다
         ↓
[3단계] 뽑아낸 정보를 Neo4j(그래프 DB)에 저장합니다
         → 점(노드)과 선(관계)으로 된 관계도가 만들어집니다
         ↓
[4단계] 사용자가 "김철수와 박영수의 관계는?" 같은 질문을 합니다
         ↓
[5단계] FastAPI + GPT가 질문을 이해하고 → Neo4j에서 답을 찾아서
         → 자연어로 답변합니다
         ↓
[6단계] Next.js(프론트엔드)가 답변과 관계도를 예쁘게 보여줍니다
```

> **세션 3~8에서 위 [2단계]~[5단계]를 직접 만듭니다.** [1단계]과 [6단계]는 이미 만들어져 있어요!

## 세션 2 — 그래프 DB 핵심: Neo4j + Cypher 기초

> **유형**: 이론 + 실습 (Neo4j Browser)
>
> 이 세션에서는 **실제로 그래프 DB를 만져봅니다**. Neo4j를 띄우고, Cypher라는 언어로 데이터를 넣고 꺼내봅니다. 세션이 끝나면 "그래프 DB 사용해 봤다"고 말할 수 있게 됩니다!

### 2.1 Neo4j란 무엇인가

Neo4j는 세계에서 가장 널리 사용되는 그래프 데이터베이스입니다. 2007년 출시 이후 금융, 보안, 추천 시스템, 생명과학 등 다양한 분야에서 활용되고 있습니다. 우리는 이 Neo4j를 사용해서 수사 관계도를 저장할 것입니다.

**Neo4j의 핵심 구성 요소:**

| 구성 요소                  | 설명                                                                  | 비유                         |
| -------------------------- | --------------------------------------------------------------------- | ---------------------------- |
| **노드 (Node)**            | 엔티티 하나. 레이블로 분류 (`:Person`, `:Location`)                   | 엑셀의 한 행(row)           |
| **관계 (Relationship)**    | 노드를 잇는 방향성 있는 연결. 타입과 속성을 가짐                      | 두 행을 잇는 화살표          |
| **속성 (Property)**        | 노드·관계에 붙는 key-value 데이터 (예: `name: '김철수'`)              | 엑셀의 셀(cell) 값          |
| **레이블 (Label)**         | 노드의 카테고리 태그. 한 노드에 여러 레이블 가능                      | 엑셀의 시트(sheet) 이름      |
| **인덱스 (Index)**         | 특정 속성에 대한 빠른 검색을 위한 자료구조                            | 책의 색인(index)             |
| **제약 조건 (Constraint)** | 데이터 무결성 보장 (예: Person의 name은 고유해야 한다)                 | 엑셀의 데이터 유효성 검사    |

**Neo4j Browser — Neo4j를 눈으로 보는 방법:**

Neo4j에는 **웹 기반 인터페이스**가 내장되어 있어서, 웹 브라우저(Chrome 등)에서 바로 사용할 수 있습니다. 쿼리를 입력하면 결과가 **그래프 형태로 시각화**되어 나타납니다.

**접속 방법:**

1. Docker로 Neo4j가 실행 중인 상태에서
2. 웹 브라우저를 열고 주소창에 `http://localhost:7474` 입력
3. 로그인 정보 입력:
   - Username: `neo4j`
   - Password: `password123` (우리가 Docker 설정에서 지정한 비밀번호)
4. 상단의 입력창에 Cypher 쿼리를 입력하고 ▶ 버튼(또는 Ctrl+Enter)으로 실행

> **localhost란?** "내 컴퓨터"를 가리키는 주소입니다. Neo4j가 내 컴퓨터에서 돌아가고 있으니, 내 컴퓨터의 7474번 포트로 접속한다는 뜻입니다.

### 2.2 Cypher란 무엇인가

**Cypher**(사이퍼)는 Neo4j에서 사용하는 **그래프 전용 쿼리 언어**입니다.

쉽게 말하면:
- MySQL에서 데이터를 다루려면 **SQL**이라는 언어를 사용하죠?
- Neo4j에서 데이터를 다루려면 **Cypher**라는 언어를 사용합니다.
- SQL이 "표(테이블)를 다루는 언어"라면, Cypher는 **"점과 선(그래프)을 다루는 언어"** 입니다.

**Cypher의 핵심 철학 — "그림을 그리듯 코딩한다":**

Cypher의 가장 큰 특징은 **쿼리가 그래프의 모양을 그대로 닮는다**는 것입니다. 마치 텍스트로 그림을 그리는 것 같습니다:

```
노드(점)를 표현:     (p:Person {name: '김철수'})
                      ↑ ↑        ↑
                      │ │ 레이블  └ 속성 (이름이 김철수)
                      │ └ Person이라는 종류
                      └ 변수명 (나중에 이 노드를 가리킬 때 사용)

관계(선)를 표현:     -[r:KNOWS {since: 2020}]->
                      ↑  ↑         ↑             ↑
                      │  │ 관계타입 └ 관계 속성   └ 화살표 (방향)
                      │  └ 변수명
                      └ 연결

전체 패턴:           (a)-[r:KNOWS]->(b)
                     "a 노드가 KNOWS 관계로 b 노드에 연결"
```

> **SQL을 아시는 분을 위한** Cypher 비교표:

| 작업       | SQL                                  | Cypher                                        |
| ---------- | ------------------------------------ | --------------------------------------------- |
| 데이터 조회 | `SELECT * FROM person`              | `MATCH (p:Person) RETURN p`                   |
| 조건 검색   | `WHERE name = '김철수'`             | `WHERE p.name = '김철수'` 또는 `{name:'김철수'}` |
| 관계 탐색   | `JOIN ... ON ...` (복잡)            | `(a)-[:KNOWS]->(b)` (직관적)                  |
| 데이터 생성 | `INSERT INTO ...`                   | `CREATE (p:Person {name: '김철수'})`          |
| 데이터 삭제 | `DELETE FROM ...`                   | `DETACH DELETE p`                             |

> **SQL을 모르셔도 됩니다!** 위 표는 참고용이고, Cypher만 배우면 됩니다.

### 2.3 Cypher 핵심 문법 — 수사 데이터로 배우기

> Neo4j Browser(`http://localhost:7474`)를 열고, 아래 쿼리를 **위에서부터 순서대로** 하나씩 복사해서 실행해 보세요.
> 각 단계가 이전 단계에서 만든 데이터를 활용합니다. 순서를 건너뛰면 안 됩니다!

**1) 노드 생성 (CREATE) — "점을 찍자"**

`CREATE`는 새로운 노드(점)를 만드는 명령어입니다. 우선 수사에 필요한 인물과 장소를 만들어 봅시다.

```cypher
// 용의자 김철수를 만듭니다
CREATE (p:Person {name: '김철수', alias: '철수', age: 35, role: '용의자'})
RETURN p
// CREATE : 새로 만들어라
// (p:Person {...}) : Person 타입의 노드를 만들고, p라는 이름을 붙여라
// {name: '김철수', ...} : 이 노드의 속성들
// RETURN p : 만든 결과를 보여줘
```

```cypher
// 용의자 박영수와 관련자 이정민도 만듭니다
CREATE (:Person {name: '박영수', age: 42, role: '용의자'})
CREATE (:Person {name: '이정민', age: 28, role: '관련자'})
```

```cypher
// 장소도 만듭니다
CREATE (:Location {name: '서울역', address: '서울시 용산구', type: '교통시설'})
CREATE (:Location {name: '강남카페', address: '서울시 강남구', type: '상업시설'})
```

> **실행 결과**: Neo4j Browser에서 동그라미(노드)가 나타납니다. 클릭하면 속성을 볼 수 있어요!
> 지금까지 만든 노드: 김철수, 박영수, 이정민, 서울역, 강남카페 — 아직은 서로 연결이 없는 점 5개입니다.

**2) 관계 생성 (MATCH + CREATE) — "점과 점을 선으로 잇자"**

이미 만들어진 노드를 찾아서(`MATCH`), 그 사이에 관계를 만듭니다(`CREATE`).

```cypher
// "김철수"와 "서울역"을 찾아서, "있었다(WAS_AT)" 관계로 연결합니다
MATCH (p:Person {name: '김철수'}), (l:Location {name: '서울역'})
CREATE (p)-[:WAS_AT {date: '2025-03-05', time: '22:00'}]->(l)
// MATCH : 이미 있는 노드를 찾아라
// (p:Person {name: '김철수'}) : 이름이 김철수인 Person을 찾아서 p라 부르자
// (l:Location {name: '서울역'}) : 이름이 서울역인 Location을 찾아서 l이라 부르자
// CREATE (p)-[:WAS_AT {...}]->(l) : p에서 l로 WAS_AT 관계를 만들어라
// 화살표(->) : 방향! "김철수가 서울역에 있었다"
```

```cypher
// 박영수도 서울역에 있었습니다 (30분 뒤에 나타남!)
MATCH (p:Person {name: '박영수'}), (l:Location {name: '서울역'})
CREATE (p)-[:WAS_AT {date: '2025-03-05', time: '22:30'}]->(l)
```

```cypher
// 인물 간의 관계도 만들어 봅시다
// 김철수는 박영수를 알고, 박영수는 이정민을 압니다
MATCH (a:Person {name: '김철수'}), (b:Person {name: '박영수'})
CREATE (a)-[:KNOWS {since: '2020'}]->(b)
```

```cypher
MATCH (a:Person {name: '박영수'}), (b:Person {name: '이정민'})
CREATE (a)-[:KNOWS {since: '2023'}]->(b)
```

> 지금까지의 그래프: 김철수 →KNOWS→ 박영수 →KNOWS→ 이정민, 그리고 김철수·박영수 →WAS_AT→ 서울역

**3) 조회 (MATCH + RETURN) — "데이터를 꺼내보자"**

```cypher
// 2025년 3월 5일에 서울역에 있었던 사람들을 찾아라
MATCH (p:Person)-[r:WAS_AT]->(l:Location {name: '서울역'})
WHERE r.date = '2025-03-05'
RETURN p.name, r.time
// (p:Person)-[r:WAS_AT]->(l:Location) : "사람이 WAS_AT 관계로 장소에 연결된" 패턴을 찾아라
// WHERE r.date = '...' : 그중에서 날짜가 3/5인 것만
// RETURN p.name, r.time : 사람 이름과 시간만 보여줘
// 결과: 김철수(22:00), 박영수(22:30) — 둘 다 서울역에 있었네요!
```

```cypher
// 김철수와 아는 사이(KNOWS)인 모든 사람을 찾아라
MATCH (a:Person {name: '김철수'})-[:KNOWS]-(b:Person)
RETURN b.name
// 여기서 화살표가 없는 - 는 "방향 상관없이"를 의미합니다
// 김철수가 아는 사람, 김철수를 아는 사람 모두 포함!
// 결과: 박영수
```

**4) 중복 방지 (MERGE) — "있으면 찾고, 없으면 만들자"**

`CREATE`는 매번 새로 만들어서 같은 데이터가 중복될 수 있습니다. `MERGE`를 사용하면 이미 있으면 기존 것을 사용하고, 없을 때만 새로 만듭니다. **실무에서는 거의 항상 MERGE를 사용합니다.**

```cypher
MERGE (p:Person {name: '김철수'})
ON CREATE SET p.age = 35, p.role = '용의자'
ON MATCH SET p.lastSeen = datetime()
RETURN p
// MERGE : 있으면 찾고, 없으면 만들어라
// ON CREATE SET : (새로 만든 경우) 이 속성들을 설정해라
// ON MATCH SET : (이미 있는 경우) 이 속성을 업데이트해라
// 이미 김철수가 있으므로 ON MATCH가 실행되어 lastSeen이 업데이트됩니다
```

**5) 경로 탐색 — 그래프 DB의 진짜 힘이 발휘되는 순간!**

이것이 그래프 DB를 쓰는 가장 큰 이유입니다. "A에서 B까지 어떻게 연결되어 있는가?"를 쉽게 찾을 수 있습니다.

```cypher
// 김철수와 이정민 사이의 최단 경로를 찾아라
// (김철수는 이정민을 직접 모르지만, 박영수를 통해 연결되어 있습니다!)
MATCH path = shortestPath(
  (a:Person {name: '김철수'})-[*]-(b:Person {name: '이정민'})
)
RETURN path
// shortestPath : 가장 짧은 연결 경로를 찾아라
// [*] : 어떤 종류의 관계든 상관없이
// 결과: 김철수 →KNOWS→ 박영수 →KNOWS→ 이정민 (그래프로 시각화됨!)
```

```cypher
// 김철수에서 3단계 이내로 연결된 모든 인물
MATCH (a:Person {name: '김철수'})-[*1..3]-(b:Person)
RETURN DISTINCT b.name
// [*1..3] : 관계 1~3개를 거쳐서 연결된
// DISTINCT : 중복 제거
// 결과: 박영수(1단계), 이정민(2단계)
```

```cypher
// KNOWS 관계만 따라서 탐색
MATCH path = (a:Person {name: '김철수'})-[:KNOWS*1..3]-(b:Person)
RETURN path
// [:KNOWS*1..3] : KNOWS 관계만, 1~3단계까지
```

**6) 집계와 분석 — "숫자로 파악하자"**

```cypher
// 가장 많은 관계를 가진 인물 (누가 핵심 인물인가?)
MATCH (p:Person)-[r]-()
RETURN p.name, COUNT(r) AS connections
ORDER BY connections DESC
// COUNT(r) : 관계의 개수를 세라
// ORDER BY connections DESC : 많은 순으로 정렬
// 결과: 박영수가 가장 많습니다! (KNOWS 2개 + WAS_AT 1개 = 3개)
// → 수사에서는 이런 "허브" 인물이 핵심입니다
```

```cypher
// 특정 장소에 가장 자주 출현한 인물
MATCH (p:Person)-[r:WAS_AT]->(l:Location)
RETURN p.name, l.name, COUNT(r) AS visits
ORDER BY visits DESC
```

**7) 삭제 — "데이터를 지우자"**

```cypher
// 노드와 연결된 모든 관계를 함께 삭제
MATCH (p:Person {name: '테스트'})
DETACH DELETE p
// DETACH : 이 노드에 연결된 관계들도 함께
// DELETE : 삭제해라
// (관계가 남아있는 노드는 삭제할 수 없어서 DETACH를 붙입니다)
```

**8) 전체 데이터 초기화 — "다음 실습을 위해 깨끗하게!"**

2.4 실습에서 새로운 시나리오를 진행할 거라, 지금까지 연습한 데이터를 지우겠습니다.

```cypher
// DB의 모든 노드와 관계를 삭제합니다 (연습용이니 과감하게!)
MATCH (n) DETACH DELETE n
// 실행 후 Neo4j Browser에서 아무것도 안 보이면 성공!
```

### 2.4 수사 미니 시나리오 실습

> 이제 배운 것을 직접 해볼 차례입니다!
> 2.3에서 전체 데이터를 초기화했으니, 깨끗한 상태에서 시작합니다.
> Neo4j Browser에서 아래 쿼리를 **순서대로** 실행해 보세요.

**시나리오 배경:**

```
인물: 김철수(용의자), 박영수(용의자), 이정민(관련자), 최지연(목격자)
장소: 서울역, 강남카페, 인천항
조직: ㈜한성물류
관계:
- 김철수 KNOWS 박영수
- 박영수 KNOWS 이정민
- 이정민 WORKS_FOR ㈜한성물류
- 박영수 OWNS ㈜한성물류
- 김철수 WAS_AT 서울역 (3/5 22:00)
- 박영수 WAS_AT 서울역 (3/5 22:30)
- 이정민 WAS_AT 강남카페 (3/5 21:00)
- 최지연 WAS_AT 서울역 (3/5 22:15)  ← 목격자
```

**Step 1: 노드 생성** (이걸 먼저 실행하세요)

```cypher
// 인물 생성
CREATE (:Person {name: '김철수', role: '용의자'})
CREATE (:Person {name: '박영수', role: '용의자'})
CREATE (:Person {name: '이정민', role: '관련자'})
CREATE (:Person {name: '최지연', role: '목격자'})

CREATE (:Location {name: '강남카페'})
CREATE (:Location {name: '인천항'})
// 조직 생성
CREATE (:Organization {name: '㈜한성물류'})
```

**Step 2: 관계 생성** (위의 노드를 만든 후 실행하세요)

```cypher
// 인물 간 관계
MATCH (a:Person {name: '김철수'}), (b:Person {name: '박영수'})
CREATE (a)-[:KNOWS]->(b)
```

```cypher
MATCH (a:Person {name: '박영수'}), (b:Person {name: '이정민'})
CREATE (a)-[:KNOWS]->(b)
```

```cypher
// 인물 ↔ 조직 관계
MATCH (p:Person {name: '이정민'}), (o:Organization {name: '㈜한성물류'})
CREATE (p)-[:WORKS_FOR]->(o)
```

```cypher
MATCH (p:Person {name: '박영수'}), (o:Organization {name: '㈜한성물류'})
CREATE (p)-[:OWNS]->(o)
```

```cypher
// 인물 ↔ 장소 관계 (출현 기록)
MATCH (p:Person {name: '김철수'}), (l:Location {name: '서울역'})
CREATE (p)-[:WAS_AT {date: '2025-03-05', time: '22:00'}]->(l)
```

```cypher
MATCH (p:Person {name: '박영수'}), (l:Location {name: '서울역'})
CREATE (p)-[:WAS_AT {date: '2025-03-05', time: '22:30'}]->(l)
```

```cypher
MATCH (p:Person {name: '이정민'}), (l:Location {name: '강남카페'})
CREATE (p)-[:WAS_AT {date: '2025-03-05', time: '21:00'}]->(l)
```

```cypher
MATCH (p:Person {name: '최지연'}), (l:Location {name: '서울역'})
CREATE (p)-[:WAS_AT {date: '2025-03-05', time: '22:15'}]->(l)
```

> **여기까지 실행하면** Neo4j Browser에서 모든 노드와 관계가 연결된 그래프를 볼 수 있습니다!
> `MATCH (n)-[r]->(m) RETURN n, r, m` 을 실행해서 전체 구조를 확인해 보세요.

**Step 3: 수사 질문에 답해 보세요!**

아래 질문을 읽고, 직접 Cypher 쿼리를 작성해 보세요. **정답은 아래쪽에 모아두었으니, 먼저 스스로 도전한 후 스크롤해서 확인하세요!**

| # | 질문 | 힌트 |
| --- | --- | --- |
| 1 | 3월 5일 서울역에 있었던 모든 사람은? | `MATCH`, `WHERE`, `RETURN` 사용 |

| 3 | 가장 많은 관계를 가진 인물은? (누가 핵심 인물인가?) | `COUNT`와 `ORDER BY` 사용 |
| 4 | 서울역에서 목격자(최지연)와 같은 시간대(±30분)에 있었던 용의자는? | 최지연의 시간은 22:15, `WHERE`에서 비교 |

**아래에 정답이 있습니다.**

---

**정답 1.** 3월 5일 서울역에 있었던 모든 사람은?

```cypher
MATCH (p:Person)-[r:WAS_AT]->(l:Location {name: '서울역'})
WHERE r.date = '2025-03-05'
RETURN p.name, p.role, r.time
ORDER BY r.time
// 결과: 김철수(22:00), 최지연(22:15), 박영수(22:30)
// → 세 명이 비슷한 시간대에 서울역에 있었습니다!
```

**정답 2.** 김철수에서 ㈜한성물류까지의 최단 경로는?

```cypher
MATCH path = shortestPath(
  (a:Person {name: '김철수'})-[*]-(b:Organization {name: '㈜한성물류'})
)
RETURN path
// 결과: 김철수 →KNOWS→ 박영수 →OWNS→ ㈜한성물류
// → 김철수는 박영수를 통해 ㈜한성물류와 연결됩니다!
```

**정답 3.** 가장 많은 관계를 가진 인물은?

```cypher
MATCH (p:Person)-[r]-()
RETURN p.name, COUNT(r) AS connections
ORDER BY connections DESC
// 결과: 박영수(4개) > 김철수(2개) > 이정민(2개) > 최지연(1개)
// → 박영수가 가장 많은 연결을 가진 핵심 인물입니다!
// 박영수: KNOWS(김철수), KNOWS(이정민), OWNS(㈜한성물류), WAS_AT(서울역)
```

**정답 4.** 서울역에서 목격자(최지연)와 같은 시간대에 있었던 용의자는?

```cypher
MATCH (witness:Person {name: '최지연'})-[rw:WAS_AT]->(l:Location {name: '서울역'}),
      (suspect:Person)-[rs:WAS_AT]->(l)
WHERE suspect.role = '용의자'
RETURN suspect.name, rs.time AS suspect_time, rw.time AS witness_time
// 결과: 김철수(22:00), 박영수(22:30) — 둘 다 목격자와 비슷한 시간대!
// → 최지연이 목격한 인물이 이 둘 중 하나(또는 둘 다)일 가능성이 높습니다
```

**1단계: Neo4j 실행 (Docker로 그래프 DB를 띄웁니다)**

### 3.3 Pydantic 모델 정의

> [라이브 코딩] | 파일: `backend/app/models.py`
>
> **Pydantic 모델이란?** API가 주고받는 데이터의 "형태"를 미리 정해놓는 것입니다.
> 예를 들어, "노드를 만들 때는 반드시 label(종류)과 properties(속성)를 보내야 한다"라고 규칙을 정해놓으면, 잘못된 데이터가 들어오는 것을 자동으로 막아줍니다.

> `backend/app/models.py`에서 만드는 것:
> - NodeCreate: 새 노드를 만들 때 필요한 데이터 형태 (label, properties)
> - RelationshipCreate: 새 관계를 만들 때 필요한 데이터 형태 (from, to, type, properties)
> - GraphResponse: 그래프 데이터를 반환할 때의 형태

### 3.4 CRUD 라우터 구현

> [라이브 코딩] | 파일: `backend/app/routers/graph.py` → `backend/app/main.py` (라우터 연결)
>
> **CRUD란?** 데이터를 다루는 4가지 기본 동작의 영어 약자입니다:
> - **C**reate (생성) — 새 데이터를 만든다
> - **R**ead (조회) — 저장된 데이터를 읽는다
> - **U**pdate (수정) — 기존 데이터를 바꾼다
> - **D**elete (삭제) — 데이터를 지운다
>
> **라우터란?** "이 URL로 요청이 오면 이 함수를 실행해라"라고 연결해주는 것입니다. 택배 분류 센터에서 "서울행은 1번 게이트, 부산행은 2번 게이트"로 보내는 것과 비슷합니다.

> `backend/app/routers/graph.py`에서 만드는 API들:
> - `POST /nodes` — 새 노드 생성 (예: 용의자 김철수 추가)
> - `GET /nodes` — 모든 노드 조회 (저장된 노드 목록 보기)
> - `POST /relationships` — 새 관계 생성 (예: 김철수-[KNOWS]->박영수)
> - `GET /full` — 전체 그래프 조회 (모든 노드와 관계)
> - `DELETE /nodes/{id}` — 특정 노드 삭제
> - `DELETE /all` — 전체 데이터 삭제
> - `GET /stats` — 통계 (노드 몇 개, 관계 몇 개)
> - `GET /duplicates` — 중복 노드 탐지 (같은 식별키를 가진 노드 확인)
> - `POST /merge-duplicates` — 중복 노드 병합 (관계 이전 후 삭제)
>
> **MERGE 동작 방식 — 노드 타입별 식별 키(Primary Key):**
>
> `MERGE`는 **모든 속성이 아닌 식별 키만**으로 기존 노드를 매칭하고, 나머지 속성은 `ON CREATE SET`/`ON MATCH SET`으로 업데이트합니다. 이렇게 하면 같은 인물이 다른 문서에서 속성이 조금 다르게 추출되더라도 중복 노드가 생기지 않습니다.
>
> | 노드 타입 | 식별 키 | 예시 |
> |-----------|---------|------|
> | Person | `name` | `MERGE (n:Person {name: '김철수'}) ON MATCH SET n.role = '용의자'` |
> | Location | `name` | |
> | Vehicle | `plate` | |
> | Phone | `number` | |
> | Account | `number` | |
> | Organization | `name` | |
> | Event | `type` | |
> | Evidence | `type` | |
>
> **주의:** Neo4j 5.x부터 `id()` 함수가 deprecated되어 `elementId()`를 사용합니다. `elementId()`는 문자열을 반환합니다.

> `backend/app/main.py`에서 하는 일:
> - lifespan: 서버 시작 시 Neo4j에 연결, 종료 시 연결 해제
> - FastAPI 앱 생성 + CORS 미들웨어 설정 (프론트엔드에서 접근 허용)
> - 라우터 등록: graph, samples 라우터를 앱에 연결
> - `GET /health`: 서버가 살아있는지 확인하는 엔드포인트

```bash

```

브라우저에서 `http://localhost:8000/docs`에 접속해 보세요. **Swagger UI**라는 API 문서 화면이 나타납니다. 여기서 우리가 만든 API들을 바로 테스트할 수 있습니다!

**Swagger UI로 API 직접 테스트하기:**

> 프론트엔드를 실행하기 전에, 우리가 만든 백엔드 API가 정말 동작하는지 **Swagger에서 직접 확인**해 봅시다.
> 각 API 카드를 펼치고 → **"Try it out"** 버튼 → 값 입력 → **"Execute"** 순서로 진행합니다.

**① 노드 생성 — `POST /graph/nodes`**

용의자 김철수를 만들어 봅시다. Request body에 아래 JSON을 입력하고 Execute:

```json
{
  "label": "Person",
  "properties": {"name": "김철수", "role": "용의자", "age": 35}
}
```

> 응답(Response body)에 `"created": [...]`가 보이면 성공! 김철수 노드가 Neo4j에 저장되었습니다.

장소도 하나 만들어 봅시다:

```json
{
  "label": "Location",
  "properties": {"name": "서울역", "type": "교통시설"}
}
```

**② 노드 조회 — `GET /graph/nodes`**

Execute만 누르면 됩니다 (입력값 없음).

> 방금 만든 김철수와 서울역이 목록에 보이나요? `labels`에 `["Person"]`, `["Location"]`이 보이고, `props`에 이름과 속성이 들어있으면 정상입니다!

**③ 관계 생성 — `POST /graph/relationships`**

김철수가 서울역에 있었다는 관계를 만듭니다:

```json
{
  "from_label": "Person",
  "from_key": "name",
  "from_value": "김철수",
  "to_label": "Location",
  "to_key": "name",
  "to_value": "서울역",
  "rel_type": "WAS_AT",
  "properties": {"date": "2025-03-05", "time": "22:00"}
}
```

> `"created": [{"type": "WAS_AT", ...}]`가 응답에 보이면 성공! 김철수→서울역 관계가 만들어졌습니다.
>
> **각 필드의 의미:**
> - `from_label`, `from_key`, `from_value`: 출발 노드를 찾는 조건 ("Person 중 name이 김철수인 노드")
> - `to_label`, `to_key`, `to_value`: 도착 노드를 찾는 조건
> - `rel_type`: 관계 종류 (WAS_AT = "있었다")
> - `properties`: 관계에 붙는 추가 정보 (날짜, 시간)

**④ 전체 그래프 조회 — `GET /graph/full`**

```json
// 응답 예시 (일부):
{
  "nodes": [
    {"id": "4:abc123:0", "labels": ["Person"], "props": {"name": "김철수", "role": "용의자", "age": 35}},
    {"id": "4:abc123:1", "labels": ["Location"], "props": {"name": "서울역", "type": "교통시설"}}
  ],
  "relationships": [
    {"source": "4:abc123:0", "target": "4:abc123:1", "type": "WAS_AT", "props": {"date": "2025-03-05", "time": "22:00"}}
  ]
}
```

> `nodes` 배열에 2개, `relationships` 배열에 1개가 보이면 완벽합니다!
> **참고:** `id`가 `"4:abc123:0"` 같은 문자열로 나오는 것이 정상입니다. Neo4j 5.x의 `elementId()` 형식입니다.

**⑤ 통계 — `GET /graph/stats`**

> `{"node_count": 2, "rel_count": 1}`이 나오면 우리 API가 모두 정상 동작하는 겁니다!

**⑥ 전체 삭제 — `DELETE /graph/all`**

> `{"message": "전체 그래프 초기화 완료"}`가 나오면 성공!

## 세션 4 — LLM 기반 정보 추출 이론 + 프롬프트 설계

> **유형**: 이론 + 실습
>
> 이 세션에서는 "AI가 텍스트를 읽고 정보를 뽑아내는 것"이 어떻게 가능한지 이해하고, 그 핵심인 **프롬프트(AI에게 주는 지시문)** 를 설계합니다.

### 4.1 LLM(대규모 언어 모델)이란 무엇인가

**ChatGPT를 써보신 적 있으시죠?**

ChatGPT에게 "이 글을 요약해줘", "이 코드의 버그를 찾아줘"라고 하면 답을 해줍니다. ChatGPT의 핵심 엔진이 바로 **LLM(Large Language Model, 대규모 언어 모델)** 입니다. 인터넷에 있는 방대한 텍스트를 학습해서 "다음에 올 말이 무엇인지"를 예측하는 AI입니다.

**그런데 이 강의에서 LLM을 왜 쓰나요?**

수사 보고서, 진술서 같은 텍스트를 생각해 보세요:

```
"목격자 이영희는 2025년 3월 5일 밤 10시경 서울역에서 김철수를 목격했다."
```

사람이 읽으면 바로 알 수 있죠:
- **누가**: 이영희 (목격자), 김철수 (목격 대상)
- **언제**: 2025년 3월 5일 밤 10시
- **어디서**: 서울역
- **무엇**: 목격

이 정보를 그래프에 넣으려면 구조화해야 합니다. 하지만 수사 보고서가 100건, 1000건이면 사람이 일일이 읽고 정리하기는 너무 힘듭니다. **LLM에게 "이 텍스트에서 인물, 장소, 관계를 뽑아줘"라고 하면 자동으로 해줍니다!**

**ChatGPT 웹사이트 vs OpenAI API — 무엇이 다른가요?**

| | ChatGPT 웹사이트 | OpenAI API |
|---|---|---|
| **사용 방법** | 브라우저에서 채팅 | 프로그램 코드에서 호출 |
| **비유** | 은행 창구에서 직접 대면 | 은행 ATM/모바일뱅킹 |
| **장점** | 편하게 대화 가능 | 자동화 가능, 프로그램에 내장 가능 |
| **이 강의에서** | 사용 안 함 | **이걸 사용합니다!** |

우리는 OpenAI **API**를 사용합니다. Python 코드에서 `openai.chat.completions.create(...)` 같은 함수를 호출하면, AI가 답변을 JSON 형태로 돌려줍니다. 이것을 우리 서버(FastAPI)가 받아서 Neo4j에 저장하는 구조입니다.

### 4.2 전통적 정보 추출 vs LLM 접근

텍스트에서 정보를 자동으로 뽑아내는 방법은 크게 두 가지가 있습니다:

**전통 방식 — 규칙/학습 기반:**

| 항목     | 설명                                                      |
| -------- | --------------------------------------------------------- |
| NER      | Named Entity Recognition — 텍스트에서 "이건 사람 이름", "이건 장소"를 자동 식별하는 기술 |
| 관계 추출 | 엔티티 쌍 사이의 관계를 분류 (예: "김철수와 서울역 → WAS_AT") |
| 장점     | 정확도 높음 (학습 데이터가 충분할 때)                     |
| 단점     | 학습 데이터를 사람이 수천~수만 건 만들어야 함. 수사 데이터 → 의료 데이터로 바꾸면 처음부터 다시 |

**LLM 방식 — 프롬프트 한 장이면 OK:**

| 항목     | 설명                                                         |
| -------- | ------------------------------------------------------------ |
| 동작     | AI에게 "이 텍스트에서 인물, 장소, 관계를 추출하라"고 지시하면 끝 |
| 장점     | 프롬프트(지시문)만 바꾸면 어떤 분야든 적용 가능. 예시 2~3개만 보여주면 정확도가 크게 향상 |
| 단점     | API 호출 비용 ($), 가끔 없는 정보를 지어내는 "환각(hallucination)" 가능 |

> **환각(Hallucination)이란?** AI가 실제로 텍스트에 없는 정보를 있는 것처럼 만들어내는 현상입니다. 예를 들어 "김철수가 인천에 갔다"는 내용이 없는데 AI가 그렇게 추출해버리는 경우입니다. 그래서 **사람이 검증하는 단계**가 중요합니다.

**우리의 접근: 하이브리드**

LLM으로 초벌 추출 → 화면에서 사람이 확인/수정 → 확정 후 그래프 저장

프론트엔드에서 LLM이 추출한 결과를 미리보기로 보여주고, 사용자가 "맞아, 저장해" 또는 "이건 틀렸어, 수정"할 수 있게 합니다.

### 4.3 트리플(Triple)이란 무엇인가

트리플은 지식 그래프의 **최소 단위**입니다. 세상의 모든 정보를 **"누가 — 무엇을 — 누구에게/어디에"** 세 조각으로 쪼갠 것입니다.

영어 문법의 **주어-동사-목적어(SVO)** 와 완전히 같은 구조입니다!

```
(Subject)  -[Predicate]->  (Object)
(주어)     -[관계/동사]->   (목적어)

일상 예시:
"철수가 영희를 안다"       → (김철수) -[KNOWS]->      (박영희)
"철수가 서울역에 있었다"    → (김철수) -[WAS_AT]->     (서울역)
"영수가 물류회사를 소유한다" → (박영수) -[OWNS]->       (㈜한성물류)
"이영희가 사건을 목격했다"   → (이영희) -[WITNESSED]->  (사건01)
```

**하나의 문서에서 여러 트리플이 추출됩니다:**

```
문서: "목격자 이영희는 3월 5일 서울역에서 김철수를 목격했다"

→ 트리플 1: (이영희) -[WITNESSED]-> (사건01)
→ 트리플 2: (이영희) -[WAS_AT]->   (서울역)
→ 트리플 3: (김철수) -[WAS_AT]->   (서울역)
```

문서 1건, 2건, 3건... 트리플이 누적될수록 그래프는 점점 풍부해지고, **처음에는 보이지 않던 연결 고리**가 드러납니다. 이것이 지식 그래프의 힘입니다!

### 4.4 트리플 추출 프롬프트 설계

**프롬프트란?** AI에게 주는 "지시문"입니다. ChatGPT에서 "이 글을 요약해줘"라고 치는 것도 프롬프트입니다. 프롬프트를 얼마나 잘 쓰느냐에 따라 AI의 답변 품질이 크게 달라집니다.

**좋은 프롬프트의 5가지 원칙:**

1. **허용 범위를 명확히**: "Person, Location, Event 중에서만 뽑아라" → AI가 엉뚱한 타입을 만들지 않음
2. **예시를 보여주기 (Few-shot)**: "이런 텍스트가 들어오면 이런 결과를 내라"는 예시 2~3개 → AI가 패턴을 학습
3. **출력 형태를 강제**: "반드시 JSON 형태로 답하라" → 프로그램이 파싱할 수 있는 형태로 받음
4. **같은 것은 하나로**: "김철수, 철수, 용의자 A가 같은 사람이면 하나로 통일하라" → 중복 방지
5. **자신감 표시**: "각 추출 결과에 확신도(0~1)를 붙여라" → 0.3이면 의심스러운 정보

> **비유**: 프롬프트 설계는 **신입사원에게 업무 매뉴얼을 만들어주는 것**과 같습니다. "이런 서류가 들어오면, 이런 형식으로 정리해서, 이 양식에 맞춰 보고해라"와 동일한 구조입니다.

**프롬프트에 포함할 내용 (세션 5에서 코드로 작성):**

- 허용된 노드 타입과 속성 목록 (Person, Location, Event, Evidence, Organization, Vehicle, Phone, Account)
- 허용된 관계 타입 목록 (KNOWS, CALLED, WAS_AT, OWNS, TRANSFERRED_TO, WITNESSED, PARTICIPATED_IN, FOUND_AT, WORKS_FOR, RELATED_TO)
- 추출 규칙 5가지 (명시 정보만 추출, alias 처리, 날짜/시간 관계 속성 포함, confidence 부여, 사실만 추출)
- JSON 출력 형식 (entities 배열 + relationships 배열, from_node/to_node 구조)

### 4.5 Structured Output 실습

> [참고] 이 세션에서 설계한 프롬프트는 세션 5에서 `backend/app/prompts/extraction.py`로 작성합니다

**Structured Output이란?** AI가 답변을 "정해진 JSON 구조"로 반환하도록 강제하는 OpenAI의 기능입니다. 이것이 없으면 AI가 "김철수는 서울역에 있었고..."처럼 자유로운 텍스트로 답하는데, 프로그램이 이걸 파싱하기 어렵습니다. Structured Output을 사용하면 반드시 `{"entities": [...], "relationships": [...]}` 같은 JSON으로 답합니다.

> OpenAI Playground(https://platform.openai.com/playground) 또는 간단한 스크립트로 직접 테스트해 봅시다.

**테스트 입력:**

```
목격자 이영희(여, 42세)는 2025년 3월 5일 밤 10시경 서울역 3번 출구 앞에서
검은색 SUV(번호판 12가 3456)에서 내린 남성 2명을 목격했다고 진술함.
한 명은 키 180cm 정도의 회색 코트 차림으로, 목격자는 이 인물이
김철수와 닮았다고 진술함.
```

**기대 출력** (AI가 이런 형태로 뽑아내면 성공):

```json
{
  "entities": [
    {"label": "Person", "properties": {"name": "이영희", "age": 42, "role": "목격자"}},
    {"label": "Person", "properties": {"name": "김철수"}},
    {"label": "Location", "properties": {"name": "서울역 3번 출구"}},
    {"label": "Vehicle", "properties": {"plate": "12가 3456", "color": "검은색", "model": "SUV"}}
  ],
  "relationships": [
    {"from": "이영희", "to": "서울역 3번 출구", "type": "WAS_AT", "properties": {"date": "2025-03-05", "time": "22:00"}},
    {"from": "김철수", "to": "서울역 3번 출구", "type": "WAS_AT", "properties": {"date": "2025-03-05", "time": "22:00"}}
  ]
}
```

> **실습 포인트**: 프롬프트를 조금씩 바꿔가며 결과가 어떻게 달라지는지 확인하세요. 이것이 **프롬프트 엔지니어링**입니다!

### 4.6 엔티티 정규화 전략

- **문제**: "김철수", "철수", "용의자 A", "회색 코트 남성" → 동일 인물?

**3단계 방어:**

1. **LLM 레벨 (프롬프트):**
   - "동일 인물로 판단되면 풀네임으로 통일하라" 지시
   - "이정민", "정민", "이 씨" → 모두 `name: "이정민"`으로 통합, 약칭은 `alias`에 기록
   - entities 배열에 같은 식별자가 두 번 나오지 않도록 명시적으로 금지

2. **DB 레벨 (MERGE 키 속성):**
   - `MERGE`가 **식별 키(name, plate, number)만**으로 매칭
   - `{name: '이정민'}`과 `{name: '이정민', alias: '서울시 관악구 거주'}`가 같은 노드로 통합됨
   - 나머지 속성은 `ON MATCH SET`으로 덮어쓰기

3. **사후 정리 (중복 병합 API):**
   - `GET /graph/duplicates`로 중복 노드 탐지
   - `POST /graph/merge-duplicates`로 관계 이전 + 속성 병합 + 중복 삭제
   - 고도화: 별도 Entity Resolution 파이프라인 + 벡터 유사도

> **프롬프트에 포함되는 핵심 규칙들:**

> 3. **관계 방향 규칙 (from → to):**
>    - 프롬프트에 허용된 관계 타입과 방향을 명시적으로 정의
>    - 예: `OWNS: Person → Vehicle/Phone/Account/Organization`, `WORKS_FOR: Person → Organization`, `WAS_AT: Person/Vehicle → Location`

## 세션 6 — GraphRAG: 자연어 질의 → 그래프 추론 → 답변

> **유형**: 실습 (라이브 코딩)
>
> 이 세션이 끝나면, **"김철수와 박영수의 관계는?"이라고 한국어로 물으면 AI가 그래프를 탐색해서 답해주는** 시스템이 완성됩니다!

### 6.1 RAG란 무엇인가 — 기본 개념부터

**먼저, 문제를 이해합시다:**

우리가 세션 5에서 Neo4j에 수사 데이터를 넣었습니다. 하지만 ChatGPT(LLM)는 이 데이터를 **전혀 모릅니다**. ChatGPT는 인터넷의 일반적인 지식만 알고 있지, "우리 Neo4j에 김철수라는 용의자가 있다"는 사실은 모릅니다.

그래서 어떻게 하냐면:

1. 사용자가 "김철수와 박영수의 관계는?"이라고 질문
2. **먼저 Neo4j에서 관련 데이터를 검색** (김철수, 박영수 관련 정보를 꺼내옴)
3. 검색 결과를 LLM에게 **"이 정보를 참고해서 답변해"** 라고 전달
4. LLM이 자연어로 답변 생성

이 패턴을 **RAG(Retrieval-Augmented Generation)** 라고 합니다. "검색(Retrieval)으로 보강(Augmented)된 생성(Generation)"이라는 뜻입니다.

> **비유**: 시험을 볼 때 **오픈 북(Open Book) 시험**과 같습니다.
> - RAG 없는 LLM = 아무것도 안 보고 기억만으로 답하는 시험 (모르는 건 틀리거나 지어냄)
> - RAG 있는 LLM = 교재를 보면서 답하는 시험 (정확한 근거를 찾아서 답함)

**전통 RAG vs GraphRAG — 뭐가 다른가요?**

|              | 전통 RAG (벡터 RAG)                    | GraphRAG (우리가 만드는 것!)                |
| ------------ | -------------------------------------- | ----------------------------------------- |
| 데이터 저장  | 문서를 쪼개서 벡터(숫자 배열)로 변환   | 데이터를 노드와 관계로 구조화              |
| 검색 방식    | "의미가 비슷한" 문서 조각을 검색       | 그래프 패턴 매칭 (Cypher 쿼리)             |
| 강점         | "비슷한 내용" 찾기에 강함              | **관계 추론**, 다단계 추적에 강함          |
| 약점         | "A와 B의 관계는?" 같은 질문 답변 불가  | 스키마 설계에 의존적                       |
| 적합한 질문  | "비슷한 사건 찾아줘"                   | "A와 B의 연결고리는?", "3단계 이내 인물은?" |

> **정리**: 수사처럼 "관계"가 핵심인 도메인에서는 GraphRAG가 훨씬 강력합니다!

**Text2Cypher — GraphRAG의 핵심 메커니즘:**

"Text2Cypher"는 이름 그대로 **텍스트(자연어)를 Cypher(그래프 쿼리)로 변환**하는 기술입니다.

사용자는 Cypher를 몰라도 됩니다. 한국어로 질문하면, AI가 자동으로 Cypher 쿼리를 만들어서 실행합니다.

```
사용자: "김철수와 박영수의 연결고리는?"
    ↓
  [1단계] AI가 이 질문을 보고, 그래프 스키마를 참고하여 Cypher를 자동 생성합니다
         → MATCH path = shortestPath(
             (a:Person {name:'김철수'})-[*]-(b:Person {name:'박영수'})
           ) RETURN path
    ↓
  [2단계] 이 Cypher를 Neo4j에서 실행 → 결과(경로 데이터)를 받음
    ↓
  [3단계] AI가 (원래 질문 + 쿼리 결과)를 함께 보고, 자연어로 답변을 생성합니다
    ↓
답변: "김철수와 박영수는 직접적 KNOWS 관계로 연결되어 있으며,
      3/5 같은 시간대에 서울역에 있었습니다."
      + 근거가 되는 서브그래프(부분 관계도) 시각화
```

> **AI가 두 번 사용됩니다!**
> 1. 첫 번째 AI 호출: 질문 → Cypher 쿼리 생성
> 2. 두 번째 AI 호출: (질문 + DB 결과) → 자연어 답변 생성

### 6.2 Cypher 생성 프롬프트 설계

> [라이브 코딩] | 파일: `backend/app/prompts/cypher_gen.py` (신규 생성)
>
> 여기서 만드는 것: **AI에게 "사용자 질문을 Cypher로 바꿔라"고 지시하는 프롬프트**

> `backend/app/prompts/cypher_gen.py`에서 만드는 프롬프트 2개:
>
> 1. **CYPHER_GENERATION_PROMPT** — "사용자의 질문을 Cypher 쿼리로 변환하라"
>    - `{schema}`: 현재 그래프에 어떤 노드/관계가 있는지 자동으로 채워짐
>    - `{question}`: 사용자의 질문이 채워짐
>    - Cypher 예시 3개를 포함해서 AI가 패턴을 학습하도록 함
>    - **추가 규칙:**
>      - 장소 검색 시 `l.name CONTAINS '서울역'` 패턴 사용 (DB에 "서울역 3번 출구", "서울역 인근 골목" 등 구체적 이름이 저장되어 있으므로)
>      - "A와 B의 관계" 질문 시 특정 관계 타입(`RELATED_TO`) 대신 **모든 관계** `-[r]-`로 탐색하고 `type(r)`, `properties(r)` 반환
>      - 직접 관계가 없으면 다중 홉 경로 탐색 (`shortestPath`)
>      - 날짜 필터링 시 `CONTAINS` 사용 (날짜 형식 불일치 대응)
>
> 2. **ANSWER_GENERATION_PROMPT** — "DB 결과를 바탕으로 자연어 답변을 생성하라"
>    - `{question}`: 원래 질문
>    - `{cypher}`: 생성된 Cypher 쿼리
>    - `{query_result}`: Neo4j 실행 결과
>    - 수사관 브리핑 톤으로 답변하도록 지시

## 세션 7 — 고도화: 그래프 알고리즘 + 수사 분석

> **유형**: 실습 (라이브 코딩)
>
> 이 세션에서는 그래프를 **수학적으로 분석**합니다. "가장 의심스러운 인물은?", "이 사건의 핵심 인물은?", "공범 그룹은?" 같은 질문에 알고리즘이 답합니다.

### 7.1 그래프 알고리즘이란 — 왜 필요한가

지금까지는 그래프에서 "김철수를 찾아줘", "두 사람의 경로를 보여줘" 같은 **특정 데이터를 검색**했습니다. 그래프 알고리즘은 한 단계 더 나아가, **그래프 전체를 보고 패턴을 발견**합니다.

> **비유**: 지금까지는 지도에서 "서울역이 어디야?"를 찾는 것이었다면, 이제는 "이 도시에서 교통의 중심지는 어디야?", "어떤 지역끼리 묶여있어?"를 분석하는 것입니다.

**중심성(Centrality) — "이 네트워크에서 누가 가장 중요한가?"**

중심성은 네트워크에서 특정 노드가 얼마나 중요한 위치에 있는지를 **수치**로 측정합니다.

| 알고리즘                   | 한마디 설명                            | 수사에서의 의미                   | 일상 비유                             |
| -------------------------- | -------------------------------------- | --------------------------------- | ------------------------------------- |
| **Degree Centrality**      | 직접 연결된 관계의 수                  | 가장 많은 관계를 가진 인물        | "인스타 팔로워가 가장 많은 사람"      |
| **Betweenness Centrality** | 다른 노드들의 최단 경로에 얼마나 끼는가 | 서로 다른 그룹을 잇는 중간 매개자 | "두 친구 그룹 모두에 끼어있는 사람" |
| **PageRank**               | "중요한 노드가 가리키는 노드"가 중요   | 영향력 순위                       | "유명인이 추천하는 맛집이 더 유명해지는 원리" |

> **수사에서 Betweenness가 특히 중요한 이유**: 범죄 조직에서 "보스"는 직접 움직이지 않습니다. 대신 **중간 매개자(브로커)**를 통해 지시합니다. Betweenness가 높은 인물이 바로 이 브로커일 가능성이 높습니다!

**커뮤니티 탐지(Community Detection) — "누가 한 패인가?"**

커뮤니티 탐지는 그래프 내에서 **서로 긴밀하게 연결된 노드 그룹**을 자동으로 식별합니다.

- 수사에서: **공범 조직의 구조**를 파악하는 데 활용
- 비유: SNS에서 "이 사람들은 서로 자주 소통하는 그룹이네" → "같은 동아리인 것 같아"를 자동으로 알아내는 것
- 대표 알고리즘: **Louvain** — 밀접하게 연결된 노드끼리 같은 색으로 묶어줌

**최단 경로(Shortest Path) — "어떻게 연결되는가?"**

두 노드 사이의 **가장 짧은 연결**을 찾습니다. 세션 2에서 배운 `shortestPath`의 확장입니다.

- 수사에서: "피해자에서 용의자까지 어떤 경로로 연결되는가?"
- 비유: 네비게이션에서 "최단 경로 찾기"와 같은 원리

**Neo4j GDS(Graph Data Science) 라이브러리:**

좋은 소식! Neo4j는 위 알고리즘들을 **내장 라이브러리(GDS)** 로 제공합니다. 복잡한 수학을 직접 구현할 필요 없이, Cypher 한 줄로 실행할 수 있습니다.

**GDS 코드 패턴 이해하기 — `analysis.py` 코드 해설:**

코드에서 GDS 알고리즘은 공통된 패턴을 따릅니다. 하나씩 뜯어보겠습니다.

**① 공통 구조 — `CALL gds.<알고리즘>.stream({...})`**

```cypher
CALL gds.betweenness.stream({
    nodeProjection: 'Person',
    relationshipProjection: {
        ALL: { type: '*', orientation: 'UNDIRECTED' }
    }
})
YIELD nodeId, score
RETURN gds.util.asNode(nodeId).name AS name, round(score, 4) AS score
ORDER BY score DESC
```

| 구성 요소 | 의미 | 비유 |
|-----------|------|------|
| `CALL gds.<알고리즘>.stream(...)` | GDS 알고리즘을 **스트림 모드**로 실행 (결과를 바로 반환) | "이 분석을 실행하고 결과를 한 줄씩 돌려줘" |
| `nodeProjection: 'Person'` | 분석 대상 **노드 라벨** 지정 — Person 노드만 분석 | "분석 대상은 '사람'만" |
| `relationshipProjection` | 분석에 사용할 **관계 설정** | "어떤 관계를 따라갈지" |
| `type: '*'` | 모든 종류의 관계를 포함 (KNOWS, CALLED, WAS_AT 등) | "종류 가리지 않고 연결된 건 다 본다" |
| `orientation: 'UNDIRECTED'` | 방향 무시 — A→B이든 B→A이든 같은 연결로 취급 | "누가 먼저 연락했든 '아는 사이'로 본다" |
| `YIELD nodeId, score` | 알고리즘 결과에서 필요한 컬럼만 꺼냄 | "결과 중 노드 ID와 점수만 가져와" |
| `gds.util.asNode(nodeId)` | 내부 노드 ID를 실제 노드 객체로 변환 | "번호표를 이름으로 바꿔줘" |

> **스트림 모드 vs 쓰기 모드**: `.stream()`은 결과를 읽기만 하고 그래프를 변경하지 않습니다. `.write()`를 쓰면 결과를 노드 속성으로 저장할 수 있지만, 이 강의에서는 실시간 조회용 `.stream()`만 사용합니다.

**② Betweenness Centrality — 매개 중심성**

```cypher
CALL gds.betweenness.stream({...})
YIELD nodeId, score
```

- **하는 일**: 모든 노드 쌍의 최단 경로를 계산하고, 특정 노드가 그 경로 위에 얼마나 자주 등장하는지 `score`로 반환
- **score가 높으면**: 이 사람을 빼면 다른 사람들 사이의 연결이 끊어진다 → **정보 브로커 / 연락책**
- **수사 활용**: score 1위 = "이 사람이 없으면 조직이 분리된다" → 검거 우선순위

**③ PageRank — 영향력 순위**

```cypher
CALL gds.pageRank.stream({...})
YIELD nodeId, score
```

- **하는 일**: Google 검색 순위의 원리 — "중요한 노드가 가리키는 노드도 중요하다"를 재귀적으로 계산
- **score(rank)가 높으면**: 많은 사람과 연결되었을 뿐 아니라, **영향력 있는 사람**과 연결된 인물
- **Degree Centrality와의 차이**: Degree는 단순히 연결 수만 셈. PageRank는 "누구와 연결되었는가"의 **질**도 고려
- **수사 활용**: "단순히 아는 사람이 많은 게 아니라, 핵심 인물들과 연결된 진짜 실세"

**④ Louvain — 커뮤니티 탐지**

```cypher
CALL gds.louvain.stream({...})
YIELD nodeId, communityId
```

- **하는 일**: 그래프에서 "서로 긴밀하게 연결된 그룹"을 자동으로 찾아서 같은 `communityId`를 부여
- **알고리즘 원리**: 모듈성(modularity)을 최대화하는 방향으로 노드를 반복적으로 그룹핑 — "그룹 내부 연결은 많고, 그룹 간 연결은 적게"
- **communityId**: 같은 번호를 받은 사람들 = 한 그룹 (번호 자체는 의미 없음, 같은지 다른지만 중요)
- **수사 활용**: 공범 조직 자동 식별 — "이 3명이 한 패, 저 2명이 한 패"

> **GDS 미설치 시**: `analysis.py`에서 GDS 호출 실패 시 `503 에러`를 반환하도록 try-except로 감싸져 있습니다. Degree Centrality(`/centrality`)만 순수 Cypher로 동작하므로 GDS 없이도 사용 가능합니다.

### 7.2 GDS 기반 분석 API 구현

> [라이브 코딩] | 파일: `backend/app/routers/analysis.py` (신규 생성) → `backend/app/main.py` (라우터 추가)
>
> 이 라우터는 **"수사 분석 도구 모음"** 입니다. 각 API가 하나의 분석 도구라고 생각하세요.

> `backend/app/routers/analysis.py`에서 만드는 API들:
>
> | API | 기능 | 쉬운 설명 |
> |-----|------|-----------|
> | `GET /centrality` | Degree 중심성 | "가장 관계가 많은 인물 순위" |
> | `GET /betweenness` | Betweenness 중심성 | "중간 매개자(브로커) 순위" |
> | `GET /pagerank` | PageRank | "영향력 순위" |
> | `GET /communities` | 커뮤니티 탐지 | "누가 한 패인가?" |
> | `GET /path/{from}/{to}` | 최단 경로 | "A에서 B까지 어떻게 연결?" |
> | `GET /common-locations` | 공통 장소 분석 | "같은 날 같은/인근 장소에 있었던 의심 쌍 (인물 + 차량)" |
> | `GET /timeline/{person}` | 행동 타임라인 | "이 사람이 언제 어디에 있었나?" |
> | `GET /suspicious-connections` | 의심 연결 | "피해자 2단계 이내 연결 인물" |
>
> **common-locations 고도화:**
> - Person뿐 아니라 **Vehicle(차량)** 노드도 탐색 대상에 포함 (차량도 `WAS_AT` 관계를 가짐)
> - **인근 장소 매칭**: "서울역 3번 출구"와 "서울역 인근 골목"처럼 정확히 같은 장소가 아니더라도 이름에 공통 키워드가 포함되면 "nearby" 매칭으로 탐지
> - 응답에 `match_type` 필드 추가: `"same_location"` (정확히 같은 장소) 또는 `"nearby"` (인근 장소)
>
> **주의:** 모든 분석 쿼리에서 `id()` 대신 `elementId()`를 사용합니다.
>
> ※ GDS 라이브러리가 설치되지 않은 환경에서는 betweenness, pagerank, communities API가 503 에러를 반환하며, 기본 Cypher 기반 centrality는 항상 동작합니다.

> `backend/app/main.py` — analysis 라우터 추가 (graph, extract, ask, analysis, samples) — **최종 완성!**

### 7.3 Swagger에서 분석 API 테스트하기

> 프론트엔드 대시보드로 넘어가기 전에, 각 분석 API의 결과를 **Swagger에서 직접 확인**해 봅시다.
> `http://localhost:8000/docs`에서 **analysis** 섹션을 펼쳐보세요.

**① 연결 중심성 — `GET /analysis/centrality`**

Execute를 누르면 **관계가 가장 많은 인물 순위**가 나옵니다:

```json
// 응답 예시:
[
  {"name": "박영수", "role": "용의자", "degree": 8},
  {"name": "김철수", "role": "용의자", "degree": 6},
  {"name": "이정민", "role": "관계인", "degree": 5},
  {"name": "최민호", "role": "피해자", "degree": 4},
  ...
]
```

> 박영수가 1위인가요? 그만큼 많은 사람/장소/조직과 연결되어 있다는 뜻입니다. **수사에서는 이런 허브 인물이 핵심입니다!**
> **참고:** `role`에 "피해자", "용의자", "목격자", "관계인" 등이 채워져 있어야 정상입니다. 비어있다면 추출 프롬프트에서 `role`이 필수로 설정되지 않은 것이므로 세션 5의 프롬프트를 확인하세요.

**② 최단 경로 — `GET /analysis/path/{from_name}/{to_name}`**

URL 파라미터에 두 인물의 이름을 입력합니다:
- `from_name`: `김철수`
- `to_name`: `최민호`

```json
// 응답 예시:
[{
  "path_nodes": ["김철수", "박영수", "최민호"],
  "path_rels": ["KNOWS", "KNOWS"],
  "hops": 2
}]
```

> 김철수와 피해자 최민호가 **2단계**로 연결되어 있군요! 중간에 박영수가 있습니다. 이것이 "경로 탐색"의 힘입니다.

**③ 공통 장소 — `GET /analysis/common-locations`**

같은 날 같은/인근 장소에 있었던 인물·차량 쌍을 자동으로 찾아줍니다:

```json
// 응답 예시:
[
  {
    "entity1": "김철수", "type1": "Person",
    "entity2": "이영희", "type2": "Person",
    "location": "서울역 3번 출구",
    "date": "2025-03-05", "time1": "22:00", "time2": "22:00",
    "match_type": "same_location"
  },
  {
    "entity1": "김철수", "type1": "Person",
    "entity2": "최민호", "type2": "Person",
    "location": "서울역 3번 출구 / 서울역 인근 골목",
    "date": "2025-03-05", "time1": "22:00", "time2": null,
    "match_type": "nearby"
  },
  {
    "entity1": "12가 3456", "type1": "Vehicle",
    "entity2": "최민호", "type2": "Person",
    "location": "서울역 3번 출구 / 서울역 인근 골목",
    "date": "2025-03-05", "time1": "21:55", "time2": null,
    "match_type": "nearby"
  }
]
```

> **`match_type`에 주목하세요!** `same_location`은 정확히 같은 장소, `nearby`는 이름에 공통 키워드("서울역")가 포함된 인근 장소입니다.
> 차량(12가 3456)도 인물과 함께 탐지되어, 사건 당일 서울역 일대에 김철수·최민호·용의 차량이 모두 있었음을 확인할 수 있습니다. **공모 탐지**의 핵심 데이터입니다.

**④ 행동 타임라인 — `GET /analysis/timeline/{person_name}`**

`person_name`에 `김철수` 입력:

```json
// 응답 예시:
[
  {"action": "WAS_AT", "target": "서울역", "date": "2025-03-05", "time": "22:00", "target_type": "Location"},
  {"action": "CALLED", "target": "박영수", "date": "2025-03-05", "time": "23:45", "target_type": "Person"}
]
```

> 김철수의 행동을 시간순으로 볼 수 있습니다. "22시에 서울역, 23:45에 박영수와 통화" — 사건 당일의 행적이 보이시나요?

**⑤ 피해자 연결 인물 — `GET /analysis/suspicious-connections`**

```json
// 응답 예시:
[
  {"victim": "최민호", "suspect": "박영수", "distance": 1, "connection_types": ["KNOWS"]},
  {"victim": "최민호", "suspect": "김철수", "distance": 2, "connection_types": ["KNOWS", "KNOWS"]}
]
```

> 피해자 최민호와 2단계 이내로 연결된 모든 인물이 표시됩니다. 가까울수록 의심도 높아집니다!

## 부록 C: 용어 사전 (Glossary)

> 강의 중 나오는 용어가 헷갈릴 때 이 표를 참고하세요. 가나다 순이 아니라 **처음 등장하는 순서**로 정리했습니다.

| 용어 | 영어 | 한 줄 설명 |
|------|------|------------|
| 데이터베이스 (DB) | Database | 데이터를 체계적으로 저장하고 꺼내는 소프트웨어 |
| 관계형 DB | RDBMS | 표(테이블) 형태로 데이터를 저장하는 전통적 DB (MySQL, PostgreSQL) |
| 그래프 DB | Graph Database | 점(노드)과 선(관계)으로 데이터를 저장하는 DB (Neo4j) |
| 노드 | Node | 그래프의 "점". 사람, 장소, 사건 등 하나의 개체를 나타냄 |
| 관계 (엣지) | Relationship / Edge | 그래프의 "선". 노드와 노드를 연결하는 관계 |
| 레이블 | Label | 노드의 종류를 나타내는 태그 (예: Person, Location) |
| 속성 | Property | 노드나 관계에 붙는 구체적 정보 (예: name: '김철수') |
| Cypher | Cypher | Neo4j에서 사용하는 그래프 전용 쿼리 언어 |
| MATCH | - | Cypher에서 "패턴을 찾아라"는 명령어 |
| CREATE | - | Cypher에서 "새로 만들어라"는 명령어 |
| MERGE | - | Cypher에서 "있으면 찾고, 없으면 만들어라"는 명령어 |
| RETURN | - | Cypher에서 "결과를 보여줘"라는 명령어 |
| 온톨로지 | Ontology | 도메인의 개념 분류 체계. "어떤 것들이 존재하고, 어떤 관계가 가능한가"의 설계도 |
| 지식 그래프 | Knowledge Graph | 온톨로지에 따라 실제 데이터를 채워 넣은 결과물 |
| 트리플 | Triple | 지식 그래프의 최소 단위. (주어)-[관계]->(목적어) |
| Docker | Docker | 프로그램을 격리된 환경(컨테이너)에서 실행하는 도구 |
| 컨테이너 | Container | Docker가 만드는 격리된 실행 환경. 가상 머신보다 가볍고 빠름 |
| FastAPI | FastAPI | Python으로 웹 서버(API)를 빠르게 만들 수 있는 프레임워크 |
| API | Application Programming Interface | 프로그램끼리 소통하는 규약. "이 URL로 이 데이터를 보내면 이 결과를 줄게" |
| REST API | - | HTTP(웹) 기반의 API 방식. URL + 메서드(GET/POST/DELETE)로 소통 |
| 엔드포인트 | Endpoint | API의 개별 주소. 예: `POST /nodes`는 "노드를 만드는 엔드포인트" |
| CRUD | Create/Read/Update/Delete | 데이터를 다루는 4가지 기본 동작 |
| Pydantic | Pydantic | Python에서 데이터의 형태(타입)를 검증해주는 라이브러리 |
| LLM | Large Language Model | 대규모 언어 모델. ChatGPT의 핵심 엔진 |
| OpenAI API | - | ChatGPT(GPT)를 프로그램에서 호출할 수 있는 인터페이스 |
| API Key | - | API를 사용하기 위한 인증 열쇠 (비밀번호와 비슷) |
| 프롬프트 | Prompt | AI에게 주는 지시문. "이 텍스트에서 인물을 추출해라" |
| Few-shot | - | AI에게 예시를 2~3개 보여줘서 패턴을 알려주는 기법 |
| Structured Output | - | AI가 정해진 JSON 구조로만 답하도록 강제하는 기능 |
| 환각 | Hallucination | AI가 실제로 없는 정보를 지어내는 현상 |
| RAG | Retrieval-Augmented Generation | AI가 외부 DB를 검색한 후 답변하는 패턴 |
| GraphRAG | - | 그래프 DB를 검색 소스로 사용하는 RAG |
| Text2Cypher | - | 자연어 질문을 Cypher 쿼리로 자동 변환하는 기술 |
| 중심성 | Centrality | 네트워크에서 노드의 중요도를 측정하는 지표 |
| PageRank | - | "중요한 노드가 가리키는 노드가 중요하다"는 원리의 순위 알고리즘 |
| 커뮤니티 탐지 | Community Detection | 그래프 내 긴밀하게 연결된 그룹을 자동 식별하는 기법 |
| GDS | Graph Data Science | Neo4j의 그래프 알고리즘 내장 라이브러리 |
| uv | - | Python 패키지 관리 도구 (pip보다 빠름) |
| CORS | Cross-Origin Resource Sharing | 다른 주소(포트)에서 API를 호출할 수 있게 허용하는 설정 |
| localhost | - | "내 컴퓨터"를 가리키는 주소. `localhost:3000` = 내 컴퓨터의 3000번 포트 |
