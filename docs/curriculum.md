## 세션 1 — 왜 그래프인가: 팔란티어가 보여준 것

> **유형**: 이론

### 1.1 데이터베이스란 무엇인가 — 그래프 DB를 이해하기 위한 출발점

데이터베이스(DB)는 데이터를 구조적으로 저장하고, 원하는 데이터를 빠르게 찾아 쓸 수 있게 해주는 소프트웨어입니다. 우리가 흔히 사용하는 데이터베이스는 크게 두 가지 철학으로 나뉩니다.

**관계형 데이터베이스 (RDBMS)**
- MySQL, PostgreSQL 같은 전통적 DB
- 데이터를 **테이블(표)** 형태로 저장: 행(row)은 개별 데이터, 열(column)은 속성
- 테이블 간 관계를 표현하려면 **JOIN** 연산을 사용해야 함
- 정형 데이터(고정된 스키마)에 강하고, 금융/회계 등에 널리 사용

**그래프 데이터베이스 (Graph DB)**
- Neo4j, Amazon Neptune 같은 DB
- 데이터를 **노드(점)** 와 **관계(선)** 으로 저장
- "누가 누구를 아는가", "어디서 누구를 만났는가" 같은 **관계 중심 질문**에 최적화
- 관계를 따라가는 탐색이 극도로 빠름 (JOIN 없이 포인터를 따라감)

**핵심 차이: 관계가 1등 시민(first-class citizen)인가?**

관계형 DB에서 "관계"는 테이블 사이의 외래 키(FK)로 간접 표현됩니다. 반면 그래프 DB에서 관계는 노드와 동등한 지위를 가진 독립적 객체입니다. 관계 자체에 속성(언제, 어디서, 어떻게)을 붙일 수 있고, 관계를 따라 탐색하는 것이 DB의 핵심 연산입니다.

### 1.2 관계형 DB의 한계와 그래프의 등장

수사관이 용의자 A의 3단계 인맥을 조회하려면?

**관계형 DB 접근:**

```sql
-- 3단계 JOIN 지옥
SELECT DISTINCT p3.name
FROM person p1
JOIN relationship r1 ON p1.id = r1.from_id
JOIN person p2 ON r1.to_id = p2.id
JOIN relationship r2 ON p2.id = r2.from_id
JOIN person p3 ON r2.to_id = p3.id
JOIN relationship r3 ON p3.id = r3.from_id
WHERE p1.name = '용의자 A';
-- N단계로 늘어나면? → 쿼리 폭발, 성능 붕괴
```

**그래프 DB 접근:**

```cypher
MATCH (a:Person {name: '용의자 A'})-[*1..3]-(connected)
RETURN DISTINCT connected.name
// 한 줄. 단계가 늘어나도 동일 구조.
```

**핵심 메시지**: 관계가 복잡하게 얽힌 데이터일수록 그래프 DB가 압도적으로 유리합니다.

**왜 이런 차이가 나는가? — Index-free Adjacency**

관계형 DB는 두 테이블을 연결하려면 매번 인덱스를 검색해야 합니다(O(log n)). 3단계 관계를 찾으려면 이 작업을 3번 반복합니다. 반면 그래프 DB는 각 노드가 인접 노드의 물리적 주소를 직접 가지고 있어, 관계를 따라가는 비용이 상수 시간(O(1))입니다. 관계가 깊어질수록 성능 격차는 기하급수적으로 벌어집니다.

| 관계 깊이 | 관계형 DB (JOIN)  | 그래프 DB (Traversal) |
| --------- | ----------------- | --------------------- |
| 1단계     | 빠름              | 빠름                  |
| 2단계     | 약간 느림         | 빠름                  |
| 3단계     | 눈에 띄게 느림    | 빠름                  |
| 5단계+    | 사실상 불가       | 여전히 빠름           |

### 1.3 팔란티어 Gotham 사례 분석

**팔란티어가 수사/정보 분석에서 하는 일:**

1. **이종 데이터 통합**: 통화 기록, 금융 거래, CCTV, 진술서, SNS → 하나의 그래프로 통합
2. **엔티티 해소 (Entity Resolution)**: "김철수" = "철수" = "용의자 A"를 동일 인물로 매핑
3. **관계 네트워크 분석**: 누가 누구와 연결되었나, 숨겨진 중간 매개자는 누구인가
4. **패턴 탐지**: 비정상적 금융 흐름, 반복적 접선 패턴, 시간대별 행동 궤적
5. **다단계 추적**: A → B → C → D로 이어지는 연결 고리에서 핵심 인물 식별

**실제 보도 사례 소개:**

- 빈 라덴 추적: 택배 기사의 통화 패턴에서 은신처 특정
- 금융 사기 네트워크: 페이퍼컴퍼니 간 자금 흐름 추적
- 마약 조직 와해: 통화/위치 데이터 그래프에서 조직 구조 도출

### 1.4 온톨로지와 지식 그래프 핵심 개념

**온톨로지(Ontology)란?**

온톨로지는 "이 세계에는 어떤 종류의 것들이 존재하고, 그것들 사이에는 어떤 관계가 가능한가"를 정의한 **설계도**입니다. 수사 도메인에서라면:
- 존재할 수 있는 것: 사람, 장소, 사건, 증거, 조직, 차량, 전화, 계좌
- 가능한 관계: 아는 사이, 통화함, 있었음, 소유함, 이체함 ...

프로그래밍으로 비유하면, 온톨로지는 **클래스(class) 정의**에 해당합니다.

**지식 그래프(Knowledge Graph)란?**

지식 그래프는 온톨로지 설계도에 따라 **실제 데이터를 채워 넣은 인스턴스**입니다. "김철수라는 Person이 서울역이라는 Location에 WAS_AT 관계로 연결"되는 것이 지식 그래프의 한 조각입니다.

프로그래밍으로 비유하면, 지식 그래프는 클래스로부터 만들어진 **객체(object) 인스턴스**들과 그 관계입니다.

**Property Graph 모델**

이 강의에서 사용하는 Neo4j는 Property Graph 모델을 채택합니다:
- **노드(Node)**: 엔티티 하나 (예: 김철수). 하나 이상의 **레이블**(예: Person)과 **속성**(예: age=35)을 가짐
- **관계(Relationship)**: 노드 사이의 연결 (예: KNOWS). **방향**이 있고, 역시 **속성**을 가질 수 있음 (예: since=2020)
- **레이블(Label)**: 노드의 분류 태그. 한 노드가 여러 레이블을 동시에 가질 수 있음

**오늘 사용할 수사 도메인 온톨로지:**

| 노드 타입      | 속성 예시                     | 설명                      |
| -------------- | ----------------------------- | ------------------------- |
| `Person`       | name, alias, age, description | 용의자, 목격자, 피해자 등 |
| `Location`     | name, address, lat, lng       | 사건 현장, 목격 장소 등   |
| `Event`        | type, date, time, description | 사건, 목격, 통화, 거래 등 |
| `Evidence`     | type, description, source     | 물증, 디지털 증거 등      |
| `Organization` | name, type                    | 회사, 단체 등             |
| `Vehicle`      | plate, model, color           | 차량 정보                 |
| `Phone`        | number, carrier               | 전화번호                  |
| `Account`      | number, bank, owner           | 금융 계좌                 |

| 관계 타입         | 설명                         |
| ----------------- | ---------------------------- |
| `KNOWS`           | 인물 간 아는 사이            |
| `CALLED`          | 전화 통화                    |
| `WAS_AT`          | 특정 시간에 장소에 있었음    |
| `OWNS`            | 소유 관계 (차량, 계좌, 전화) |
| `TRANSFERRED_TO`  | 금전 이체                    |
| `WITNESSED`       | 목격                         |
| `PARTICIPATED_IN` | 사건 참여                    |
| `FOUND_AT`        | 증거 발견 장소               |
| `WORKS_FOR`       | 소속 조직                    |
| `RELATED_TO`      | 기타 관련성                  |

## 세션 2 — 그래프 DB 핵심: Neo4j + Cypher 기초

> **유형**: 이론 + 실습 (Neo4j Browser)

### 2.1 Neo4j란 무엇인가

Neo4j는 세계에서 가장 널리 사용되는 그래프 데이터베이스입니다. 2007년 출시 이후 금융, 보안, 추천 시스템, 생명과학 등 다양한 분야에서 활용되고 있습니다.

**Neo4j의 핵심 구성 요소:**

| 구성 요소                  | 설명                                                                  | 비유                         |
| -------------------------- | --------------------------------------------------------------------- | ---------------------------- |
| **노드 (Node)**            | 엔티티 하나. 레이블로 분류 (`:Person`, `:Location`)                   | 엑셀의 한 행(row)           |
| **관계 (Relationship)**    | 노드를 잇는 방향성 있는 연결. 타입과 속성을 가짐                      | 두 행을 잇는 화살표          |
| **속성 (Property)**        | 노드·관계에 붙는 key-value 데이터 (예: `name: '김철수'`)              | 엑셀의 셀(cell) 값          |
| **레이블 (Label)**         | 노드의 카테고리 태그. 한 노드에 여러 레이블 가능                      | 엑셀의 시트(sheet) 이름      |
| **인덱스 (Index)**         | 특정 속성에 대한 빠른 검색을 위한 자료구조                            | 책의 색인(index)             |
| **제약 조건 (Constraint)** | 데이터 무결성 보장 (예: Person의 name은 고유해야 한다)                 | 엑셀의 데이터 유효성 검사    |

**Neo4j Browser**: Neo4j에 내장된 웹 기반 인터페이스로, Cypher 쿼리를 실행하고 결과를 그래프로 시각화할 수 있습니다. `http://localhost:7474`에서 접근합니다.

### 2.2 Cypher란 무엇인가

Cypher는 Neo4j에서 사용하는 **그래프 전용 쿼리 언어**입니다. SQL이 관계형 DB를 다루는 언어라면, Cypher는 그래프 DB를 다루는 언어입니다.

**Cypher의 핵심 철학 — ASCII Art 패턴 매칭:**

Cypher의 가장 큰 특징은 쿼리가 그래프의 모양을 그대로 닮는다는 것입니다:

```
노드:   (변수:레이블 {속성})        → (p:Person {name: '김철수'})
관계:   -[변수:타입 {속성}]->       → -[r:KNOWS {since: 2020}]->
경로:   (a)-[r]->(b)                → 노드a에서 노드b로의 관계r
```

SQL과 Cypher 비교:

| 작업       | SQL                                  | Cypher                                        |
| ---------- | ------------------------------------ | --------------------------------------------- |
| 데이터 조회 | `SELECT * FROM person`              | `MATCH (p:Person) RETURN p`                   |
| 조건 검색   | `WHERE name = '김철수'`             | `WHERE p.name = '김철수'` 또는 `{name:'김철수'}` |
| 관계 탐색   | `JOIN ... ON ...` (복잡)            | `(a)-[:KNOWS]->(b)` (직관적)                  |
| 데이터 생성 | `INSERT INTO ...`                   | `CREATE (p:Person {name: '김철수'})`          |
| 데이터 삭제 | `DELETE FROM ...`                   | `DETACH DELETE p`                             |

### 2.3 Cypher 핵심 문법 — 수사 데이터로 배우기

> Neo4j Browser에서 직접 실행하며 학습

**노드 생성 (CREATE)**

```cypher
// 용의자 생성
CREATE (p:Person {name: '김철수', alias: '철수', age: 35, role: '용의자'})
RETURN p

// 장소 생성
CREATE (l:Location {name: '서울역', address: '서울시 용산구', type: '교통시설'})
RETURN l
```

**관계 생성 (CREATE + MATCH)**

```cypher
// 김철수가 서울역에 있었다
MATCH (p:Person {name: '김철수'}), (l:Location {name: '서울역'})
CREATE (p)-[:WAS_AT {date: '2025-03-05', time: '22:00'}]->(l)
```

**조회 (MATCH + WHERE)**

```cypher
// 특정 날짜에 서울역에 있었던 사람들
MATCH (p:Person)-[r:WAS_AT]->(l:Location {name: '서울역'})
WHERE r.date = '2025-03-05'
RETURN p.name, r.time

// 김철수와 아는 사이인 모든 사람
MATCH (a:Person {name: '김철수'})-[:KNOWS]-(b:Person)
RETURN b.name
```

**중복 방지 (MERGE)**

```cypher
// 이미 있으면 매칭, 없으면 생성
MERGE (p:Person {name: '김철수'})
ON CREATE SET p.age = 35, p.role = '용의자'
ON MATCH SET p.lastSeen = datetime()
RETURN p
```

**경로 탐색 — 그래프 DB의 진짜 힘**

```cypher
// 김철수와 박영수 사이의 최단 경로
MATCH path = shortestPath(
  (a:Person {name: '김철수'})-[*]-(b:Person {name: '박영수'})
)
RETURN path

// 3단계 이내 연결된 모든 인물
MATCH (a:Person {name: '김철수'})-[*1..3]-(b:Person)
RETURN DISTINCT b.name

// 특정 관계만 따라가기
MATCH path = (a:Person {name: '김철수'})-[:KNOWS|CALLED*1..3]-(b:Person)
RETURN path
```

**집계와 분석**

```cypher
// 가장 많은 관계를 가진 인물 (연결 중심성)
MATCH (p:Person)-[r]-()
RETURN p.name, COUNT(r) AS connections
ORDER BY connections DESC
LIMIT 5

// 특정 장소에 가장 자주 출현한 인물
MATCH (p:Person)-[r:WAS_AT]->(l:Location)
RETURN p.name, l.name, COUNT(r) AS visits
ORDER BY visits DESC
```

**삭제**

```cypher
// 노드와 연결된 모든 관계 삭제
MATCH (p:Person {name: '테스트'})
DETACH DELETE p
```

## 세션 4 — LLM 기반 정보 추출 이론 + 프롬프트 설계

> **유형**: 이론 + 실습

### 4.1 LLM(대규모 언어 모델)이란 무엇인가

LLM(Large Language Model)은 방대한 텍스트 데이터로 학습된 AI 모델로, 자연어를 이해하고 생성할 수 있습니다. GPT-4, Claude 등이 대표적입니다.

**이 강의에서 LLM을 쓰는 이유:**

비정형 텍스트(수사 보고서, 진술서 등)에서 구조화된 정보(누가, 언제, 어디서, 무엇을)를 자동으로 뽑아내야 합니다. 전통적으로는 사람이 직접 읽고 정리하거나, 규칙 기반 프로그램(정규표현식 등)을 만들어야 했습니다. LLM은 프롬프트(지시문) 하나로 이 작업을 수행할 수 있습니다.

### 4.2 전통적 정보 추출 vs LLM 접근

**전통 방식 (NER + 관계 추출):**

| 항목     | 설명                                                      |
| -------- | --------------------------------------------------------- |
| NER      | Named Entity Recognition — 텍스트에서 사람, 장소 등을 식별 |
| 관계 추출 | 엔티티 쌍 사이의 관계를 분류                              |
| 장점     | 정확도 높음 (학습 데이터가 충분할 때)                     |
| 단점     | 학습 데이터 필요, 도메인 변경 시 재학습, 구축 비용 큼     |

**LLM 방식:**

| 항목     | 설명                                                         |
| -------- | ------------------------------------------------------------ |
| 동작     | 프롬프트에 "이 텍스트에서 인물, 장소, 관계를 추출하라"고 지시 |
| 장점     | 프롬프트만으로 도메인 전환, Few-shot 예시로 정확도 향상, Structured Output으로 안정적 파싱 |
| 단점     | API 비용, 응답 속도, 환각(hallucination) 가능성              |

**하이브리드가 최적**: LLM으로 초벌 추출 → 사람이 검증 → 확정 후 그래프 저장

### 4.3 트리플(Triple)이란 무엇인가

트리플은 지식 그래프의 최소 단위로, **주어-술어-목적어** 구조입니다. 모든 지식을 이 세 요소로 분해합니다.

```
(Subject)  -[Predicate]->  (Object)
(주어)     -[관계]->       (목적어)

예시:
(김철수)   -[WAS_AT]->     (서울역)     → "김철수는 서울역에 있었다"
(박영수)   -[OWNS]->       (㈜한성물류)  → "박영수는 ㈜한성물류를 소유한다"
(이영희)   -[WITNESSED]->  (사건01)     → "이영희는 사건01을 목격했다"
```

하나의 문서에서 여러 트리플이 추출되고, 이 트리플들이 모여 지식 그래프를 형성합니다. 트리플이 누적될수록 그래프는 점점 풍부해지고, 숨겨진 관계가 드러납니다.

### 4.4 트리플 추출 프롬프트 설계

**프롬프트 설계 핵심 원칙:**

1. **온톨로지 스키마를 명시적으로 주입**: 허용된 노드/관계 타입을 프롬프트에 포함
2. **Few-shot 예시 제공**: 입력 텍스트 → 기대 출력 JSON 쌍을 2~3개
3. **출력 포맷 강제**: JSON mode 또는 Structured Output 사용
4. **엔티티 정규화 지시**: "동일 인물은 하나의 이름으로 통일하라"
5. **불확실성 표현**: 확신도(confidence) 필드 포함

**프롬프트 설계 내용 (세션 5에서 `backend/app/prompts/extraction.py`로 작성):**

- 허용된 노드 타입과 속성 목록 (Person, Location, Event, Evidence, Organization, Vehicle, Phone, Account)
- 허용된 관계 타입 목록 (KNOWS, CALLED, WAS_AT, OWNS, TRANSFERRED_TO, WITNESSED, PARTICIPATED_IN, FOUND_AT, WORKS_FOR, RELATED_TO)
- 추출 규칙 5가지 (명시 정보만 추출, alias 처리, 날짜/시간 관계 속성 포함, confidence 부여, 사실만 추출)
- JSON 출력 형식 (entities 배열 + relationships 배열, from_node/to_node 구조)

### 4.5 Structured Output 실습

> [참고] 이 세션에서 설계한 프롬프트는 세션 5에서 `backend/app/prompts/extraction.py`로 작성합니다**

## 세션 6 — GraphRAG: 자연어 질의 → 그래프 추론 → 답변

> **유형**: 실습 (라이브 코딩)

### 6.1 RAG란 무엇인가 — 기본 개념부터

**RAG(Retrieval-Augmented Generation)** 는 LLM이 답변을 생성할 때, 외부 데이터를 먼저 검색(Retrieval)한 뒤 그 결과를 참고하여 답변을 만드는 패턴입니다.

LLM은 학습 시점의 데이터만 알고 있으므로, 우리가 방금 만든 수사 지식 그래프의 내용은 모릅니다. RAG를 통해 "먼저 DB에서 관련 정보를 찾고, 그 정보를 바탕으로 답변하라"고 지시합니다.

**전통 RAG vs GraphRAG:**

|              | 전통 RAG (벡터 RAG)                    | GraphRAG                                  |
| ------------ | -------------------------------------- | ----------------------------------------- |
| 데이터 저장  | 문서를 쪼개서 벡터(숫자 배열)로 변환   | 데이터를 노드와 관계로 구조화              |
| 검색 방식    | "의미가 비슷한" 문서 조각을 벡터 유사도로 검색 | 그래프 패턴 매칭 (Cypher 쿼리)     |
| 강점         | 의미적 유사 문서 검색에 강함           | **관계 추론**, 다단계 추적에 강함          |
| 약점         | "A와 B의 관계는?" 같은 관계 추론 불가  | 스키마 설계에 의존적                       |
| 적합한 질문  | "비슷한 사건 찾아줘"                   | "A와 B의 연결고리는?", "3단계 이내 인물은?" |

**Text2Cypher — GraphRAG의 핵심 메커니즘:**

Text2Cypher는 사용자의 자연어 질문을 그래프 DB가 이해할 수 있는 Cypher 쿼리로 변환하는 기술입니다. SQL 세계에서 Text2SQL(자연어→SQL)이 있듯이, 그래프 세계에서는 Text2Cypher가 있습니다.

```
사용자: "김철수와 박영수의 연결고리는?"
    ↓ [1단계] LLM이 그래프 스키마를 참고하여 Cypher 생성
    ↓  → MATCH path = shortestPath((a:Person {name:'김철수'})-[*]-(b:Person {name:'박영수'})) RETURN path
    ↓ [2단계] Neo4j에서 Cypher 실행 → 결과 반환
    ↓ [3단계] LLM이 (질문 + 쿼리 결과)를 받아 자연어 답변 생성
    ↓
답변: "김철수와 박영수는 직접적 KNOWS 관계로 연결되어 있으며, 3/5 같은 시간대에 서울역에 있었습니다."
      + 근거 서브그래프 시각화
```

### 6.2 Cypher 생성 프롬프트 설계

> [라이브 코딩] | 파일: `backend/app/prompts/cypher_gen.py` (신규 생성)

> `backend/app/prompts/cypher_gen.py` — CYPHER_GENERATION_PROMPT ({schema}, {question} 플레이스홀더, Cypher 예시 3개) / ANSWER_GENERATION_PROMPT ({question}, {cypher}, {query_result} 플레이스홀더, 수사관 브리핑 톤)

### 6.3 질의응답 엔드포인트 구현

> [라이브 코딩] | 파일: `backend/app/routers/ask.py` (신규 생성) → `backend/app/main.py` (라우터 추가)

## 세션 7 — 고도화: 그래프 알고리즘 + 수사 분석

> **유형**: 실습 (라이브 코딩)

### 7.1 그래프 알고리즘이란 — 왜 필요한가

지금까지는 그래프에서 특정 노드를 찾거나 경로를 조회하는 **검색**을 했습니다. 그래프 알고리즘은 한 단계 더 나아가, 그래프 전체의 **구조적 특성**을 수학적으로 분석합니다. "누가 이 네트워크에서 가장 중요한 사람인가?", "이 네트워크에 숨겨진 그룹이 있는가?" 같은 질문에 답할 수 있습니다.

**중심성(Centrality) — "누가 중요한가?"**

중심성은 네트워크에서 특정 노드가 얼마나 중요한 위치에 있는지를 측정하는 지표입니다.

| 알고리즘                   | 측정 기준                              | 수사 활용                         | 쉬운 비유                             |
| -------------------------- | -------------------------------------- | --------------------------------- | ------------------------------------- |
| **Degree Centrality**      | 직접 연결된 관계의 수                  | 가장 많은 관계를 가진 인물        | "친구가 가장 많은 사람"               |
| **Betweenness Centrality** | 다른 노드들의 최단 경로에 얼마나 끼는가 | 서로 다른 그룹을 잇는 중간 매개자 | "두 그룹 사이에서 다리 역할하는 사람" |
| **PageRank**               | "중요한 노드가 가리키는 노드"가 중요   | 영향력 순위                       | "유명인이 추천하는 맛집이 더 유명"    |

**커뮤니티 탐지(Community Detection) — "누가 한 패인가?"**

커뮤니티 탐지는 그래프 내에서 서로 긴밀하게 연결된 노드 그룹을 자동으로 식별합니다. 수사에서는 공범 조직의 구조를 파악하는 데 활용됩니다. Louvain 알고리즘이 대표적이며, 모듈성(modularity)을 최대화하는 방향으로 그룹을 나눕니다.

**최단 경로(Shortest Path) — "어떻게 연결되는가?"**

두 노드 사이의 최소 단계 연결을 찾습니다. "피해자에서 용의자까지 어떤 경로로 연결되는가?"를 추적할 때 핵심적으로 사용됩니다.

**Neo4j GDS(Graph Data Science) 라이브러리:**

Neo4j는 이런 그래프 알고리즘을 내장 라이브러리(GDS)로 제공합니다. GDS를 사용하면 별도 코드 없이 Cypher 한 줄로 복잡한 알고리즘을 실행할 수 있습니다.

### 7.2 GDS 기반 분석 API 구현

> [라이브 코딩] | 파일: `backend/app/routers/analysis.py` (신규 생성) → `backend/app/main.py` (라우터 추가)

> `backend/app/routers/analysis.py` — GET /centrality (Cypher 기반 degree), GET /betweenness (GDS, 503 fallback), GET /pagerank (GDS, 503 fallback), GET /communities (GDS Louvain, 503 fallback), GET /path/{from}/{to} (shortestPath), GET /common-locations (같은 날 같은 장소 인물 쌍), GET /timeline/{person} (시간순 행동 궤적), GET /suspicious-connections (피해자 2단계 연결)

> `backend/app/main.py` — analysis 라우터 추가 (graph, extract, ask, analysis, samples) — 최종 완성

### 7.3 프론트엔드에서 분석 결과 확인

**프론트엔드 분석 대시보드(`http://localhost:3000/analysis`)에서 제공하는 화면:**

- **중심성 대시보드**: 인물별 연결 수, PageRank 순위, 매개 중심성 — 막대 그래프
- **커뮤니티 뷰**: 같은 커뮤니티 인물들을 같은 색으로 표시한 그래프
- **타임라인 뷰**: 특정 인물의 시간순 행동 궤적
- **경로 탐색**: 두 인물 선택 → 최단 경로 하이라이트

**수강생 실습:**

- "가장 많은 관계를 가진 인물은?" → 중심성 분석
- "김철수와 최민호(피해자)를 연결하는 경로는?" → 경로 탐색
- "같은 날 같은 장소에 있었던 인물 쌍은?" → 공모 탐지

> [뒤처진 경우] `cp -r backend/checkpoints/session7_complete/app/* backend/app/`

